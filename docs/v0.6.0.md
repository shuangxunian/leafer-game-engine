# leaferGame v0.6.0

## Version Positioning

`v0.6.0` starts the next stage:

**Runtime Tooling Panel Sprint**

The previous stages made runtime state inspectable, introduced data-driven content primitives, and migrated part of the example to data-driven setup. This stage focuses on turning those data sources into a more useful runtime tooling surface.

---

## Why This Stage

`0.4.x` introduced debug snapshots, browser overlay, collider visualization, and inspector snapshots.

`0.5.x` introduced asset manifests, entity templates, scene config, and component schemas.

The next bottleneck is presentation and workflow:

- `BrowserDebugOverlay` is useful but compact and debug-only
- `ToolingSnapshot` can aggregate debug and inspector data, but does not yet have a good presentation format
- component schemas are available but not visible to tooling users yet
- scene/entity data exists but is not easy to scan during a live example
- future editor work needs a better runtime panel foundation

`0.6.x` should make runtime tooling more readable and composable without jumping straight to a full editor.

---

## Stage Goals

`0.6.x` should move the project toward:

- richer formatting for debug and inspector data
- browser-visible tooling panel structure
- clearer separation between data formatting and DOM presentation
- optional sections for debug, inspector, assets, systems, and schema metadata
- example integration that can show more than FPS/entity count
- a foundation for future interactive inspector panels

---

## Current Priority

The first useful step is `v0.6.1`: tooling snapshot formatting.

Reason:

- formatting can be tested without a browser DOM
- `BrowserDebugOverlay` already uses pure formatting helpers
- later panels can reuse the same formatted sections
- it keeps presentation separate from runtime data collection

---

## Scope Boundaries

This stage should include:

- pure formatting helpers
- lightweight browser panel classes
- tests for formatter stability
- example integration where useful

This stage should not include:

- visual editor UI
- runtime mutation of components
- drag-and-drop hierarchy editing
- persistent editor layout
- remote devtools integration

---

## Done When

By the end of `0.6.x`:

- a tooling snapshot can be formatted as a readable panel
- a browser panel can show debug and inspector data
- entity/component summaries are visible during the example
- schema metadata can be surfaced in tooling output
- `npm run check`, `npm test`, and `npm run build:example` pass

---

## Relation to Previous Stages

- `0.4.x` made runtime state easier to inspect.
- `0.5.x` made content easier to describe and validate.
- `0.6.x` should make runtime tooling easier to read and use while building content.
