# leaferGame v0.0.2

## 版本定位

`v0.0.2` 不追求新增很多玩法能力。

这一版的目标是：

- 把当前可运行原型收紧成一个更稳定的引擎骨架
- 先解决生命周期、运行时循环、实体安全增删、销毁边界、基础回归测试
- 为后续 `v0.0.3` 的 framework 复用能力打底

可以把它理解为一次：

**Stability Sprint**

---

## 版本目标

`v0.0.2` 完成后，项目应该达到下面这些状态：

- `Scene` / `System` 生命周期不会重复启动
- `BrowserRuntime` 在任何时候都只会存在一个主循环
- `World` 在 update 过程中增删实体仍然安全
- destroy 语义明确且尽量幂等
- 核心生命周期具备最小自动化回归保护

---

## 范围边界

这一版要做：

- core 生命周期稳定性
- runtime 运行控制
- world 安全增删
- 销毁责任划分
- 最小测试基线

这一版不做：

- 真正可用的 camera 体系
- 完整资源加载与 preload
- collision `enter/stay/exit`
- prefab / factory 抽象
- 编辑器能力和更重的 tooling

---

## 任务清单

### 1. 收紧 `Scene` / `System` 生命周期

目标：

- 保证每个 system 在一个 scene 生命周期内只 `start()` 一次

具体任务：

- 调整 `Scene.start()` 与 `Scene.addSystem()` 的启动顺序
- 或在 `System` 内增加内部启动保护，避免重复初始化
- 明确 `onStart()` 阶段新增 system 的行为
- 明确 scene destroy 后再次使用是否允许；如果不允许，要在语义上写清楚

涉及位置：

- `src/core/scene.ts`
- `src/core/system.ts`

完成标准：

- 在 `onStart()` 内 `addSystem()` 不会导致重复 `start()`
- scene 启动后动态追加 system 仍然只初始化一次
- 不会因为重复调用 `start()` 产生双重副作用

---

### 2. 防止重复 runtime 主循环

目标：

- 任意时刻只存在一个 `requestAnimationFrame` 循环

具体任务：

- 给 `BrowserRuntime` 增加 `isRunning` 状态
- 在重复调用 `start()` 时做保护或明确切换语义
- `stop()` 时统一清理 RAF id 和运行态
- 明确 scene 切换时 runtime 与 scene 的关系

涉及位置：

- `src/runtime/browser-runtime.ts`
- `src/core/game.ts`

完成标准：

- 重复调用 `start()` 不会叠加出多个主循环
- `stop()` 之后不再继续 tick
- stop 之后再次 start 可以正常恢复

---

### 3. 给 `World` 增加安全增删队列

目标：

- 在 update / fixedUpdate / lateUpdate 期间增删实体不会破坏遍历

具体任务：

- 为 entity create / destroy 增加 pending queue
- 统一在确定的帧边界 flush
- 让重复 destroy 请求安全化
- 明确实体是在“本帧生效”还是“下一帧生效”

涉及位置：

- `src/core/world.ts`
- `src/core/entity.ts`
- 可能需要配合 `src/core/scene.ts` 或 `src/core/game.ts`

完成标准：

- system 或 component 在 update 期间创建实体不会打乱当前遍历
- system 或 component 在 update 期间销毁实体不会导致漏更新或重复更新
- repeated destroy 不会抛错或产生不一致状态

当前约定：

- update 期间新建的 entity 在本次外层 phase 结束后进入 `world.entities`
- update 期间销毁的 entity 会立刻失活，并在本次外层 phase 结束后真正移除

---

### 4. 明确 destroy 语义和 ownership

目标：

- 让“谁负责销毁谁”这件事变得清楚且可重复调用

具体任务：

- 约定 `Scene`、`World`、`Entity`、`Component`、`System`、`RenderNode` 的 ownership
- 让高风险 destroy 路径尽量幂等
- 检查渲染节点、事件监听、桥接对象是否有明确释放点
- 补文档说明场景切换时哪些对象应该自动销毁，哪些由调用方负责

涉及位置：

- `src/core/scene.ts`
- `src/core/world.ts`
- `src/core/entity.ts`
- `src/framework/view.ts`
- `src/framework/keyboard.ts`
- `examples/dodge-blocks/boot.ts`
- `docs/architecture.md`

完成标准：

- 重复 destroy 不会产生明显副作用
- scene 切换后不残留旧渲染节点
- 输入桥接这类运行时副作用有明确释放策略

当前约定：

- `Scene.destroy()` 后视为终态，不允许再次 `start()`
- `runtime.stop()` 只停止主循环，不自动销毁 active scene
- `runtime.start(newScene)` 会替换并销毁旧 scene

---

### 5. 建立最小回归测试基线

目标：

- 把这一版最容易回归的生命周期问题先自动化兜住

具体任务：

- 为 `Game.tick()` 增加基础行为测试
- 为 `Scene.start()` 增加生命周期测试
- 为单个 system 只启动一次增加测试
- 为 runtime 的 start / stop / repeated start 增加测试
- 为 world 的 add / remove queue 增加测试

建议覆盖点：

- scene set / scene replace
- update / fixedUpdate / lateUpdate 调用顺序
- `onStart()` 期间 addSystem
- update 中创建和销毁 entity

建议位置：

- `src/core/__tests__/` 或 `tests/core/`
- `src/runtime/__tests__/` 或 `tests/runtime/`

完成标准：

- 本文档列出的核心风险都有对应自动化测试
- 后续重构 core/runtime 时可以快速发现回归

---

## 建议执行顺序

建议按下面顺序推进，不要并行打散：

1. 先修 `Scene` / `System` 生命周期
2. 再修 runtime 单循环
3. 然后实现 `World` 安全增删队列
4. 接着补 destroy 语义与示例清理点
5. 最后补测试并回头收口 API 行为

原因：

- 前两项决定主生命周期是否可靠
- 第三项决定后续 framework 能不能放心扩展
- 第四项决定 scene 切换和 sample 是否会埋脏状态
- 第五项负责把前面成果固化下来

## 交付物

`v0.0.2` 结束时建议至少产出这些内容：

- 一版更稳定的 core/runtime 实现
- 一组可运行的生命周期回归测试
- 一份更新后的架构说明，明确 destroy 与 ownership
- 一个仍可正常运行的 sample，作为回归验证对象

---

## 验收清单

发布 `v0.0.2` 前，至少确认下面这些问题答案都是“是”：

- system 是否保证只启动一次
- runtime 是否保证只跑一个主循环
- stop 后是否完全停止 ticking
- update 中增删 entity 是否安全
- scene 切换后是否不会遗留旧状态
- 核心生命周期是否已有自动化测试覆盖

---

## 与后续版本的关系

`v0.0.2` 完成后，下一版建议进入：

**`v0.2.0` = Reusability Sprint**

优先处理：

- `System` priority / ordering
- `World` query API
- component dependency validation
- render layer 拆分
- 可复用状态流抽象

也就是说：

`v0.0.2` 先把“原型骨架站稳”，`v0.2.0` 再把“复用能力做厚”。
