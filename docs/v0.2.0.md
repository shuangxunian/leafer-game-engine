# leaferGame v0.2.0

## 版本命名说明

后续版本不再继续沿用 `v0.0.x` 这种只适合 very early prototype 的命名。

原因很简单：

- npm 包当前已经是 `0.1.0`
- `v0.0.2` 这一轮已经不只是随手试验，而是在系统性地收紧引擎骨架
- 下一轮的目标也不是修一个小补丁，而是把项目从“稳定骨架”推进到“可复用框架”

所以，下一阶段更适合命名为：

**`v0.2.0`**

这个版本代表的不是“已经成熟”，而是：

- 项目还处在 `0.x` 阶段
- 但正在进入一个更清晰的 milestone 节奏
- 每个 minor 版本对应一个明确阶段目标

建议后续按下面的节奏理解：

- `0.1.x`
  - 稳定骨架与生命周期收口
- `0.2.x`
  - 框架复用能力成形
- `0.3.x`
  - engine-facing 能力扩张
- `0.4.x`
  - tooling / editor-friendly 结构增强

---

## 版本定位

`v0.2.0` 是：

**Reusability Sprint**

建议拆成两个 patch 批次推进：

- [v0.2.1](/Users/shuangxunian/code/leaferGame/docs/v0.2.1.md)
  - `System` priority / ordering
  - `World` query API
  - component dependency validation
- [v0.2.2](/Users/shuangxunian/code/leaferGame/docs/v0.2.2.md)
  - render layer 拆分
  - sample layer 接入
  - 轻量状态流抽象

这一版的核心不是继续修底层稳定性，也不是扩 demo 内容，而是让当前引擎开始具备真正“能复用到第二个 sample”的结构能力。

如果说 `v0.0.2` / `0.1.x` 做的是：

- 把骨架站稳

那么 `v0.2.0` 要做的是：

- 把骨架变成框架

---

## 版本目标

`v0.2.0` 完成后，项目应该至少具备下面这些特征：

- system 执行顺序是显式的、稳定的
- world 具备基础 query API
- 常见 component 组合的依赖关系能被尽早发现
- render 层具备基本层次结构，而不是所有节点都塞进同一个 root
- sample 中的通用状态流不再完全散落在 example 代码里

这意味着：

- 第二个 sample 接入时，不需要重新发明一套调度和查找方式
- framework 层开始真正变厚
- examples 会更像“消费者”，而不是“补 engine 空缺的地方”

---

## 范围边界

这一版要做：

- `System` priority / ordering
- `World` query API
- component dependency validation
- render layer 拆分
- 可复用状态流抽象

这一版不做：

- collision `enter/stay/exit`
- spatial partition
- 真正可用的 camera 跟随/缩放体系
- 资源 preload/cache 正式化
- prefab / template 数据化
- editor/tooling 扩展

## 任务清单

### 1. 增加 `System` priority / ordering

目标：

- 不再依赖“谁先注册谁先执行”的隐式顺序

具体任务：

- 给 `System` 增加 priority 字段
- 在 scene 内维护稳定排序
- 明确同优先级 system 的相对顺序策略
- 让 `update` / `fixedUpdate` / `lateUpdate` 都遵守同一套顺序

涉及位置：

- `src/core/system.ts`
- `src/core/scene.ts`

完成标准：

- input、movement、collision、view sync 可以按预期顺序执行
- 新增 system 时不会因为注册时机不同而出现隐式行为漂移

---

### 2. 增加 `World` query API

目标：

- 让系统不再手写全量扫描和重复过滤逻辑

具体任务：

- 增加 `getEntitiesWith`
- 增加 `getEntitiesWithAll`
- 增加 `getEntitiesWithAny`
- 明确 query 是否返回 live snapshot
- 保证 query 语义与 pending add/remove queue 一致

涉及位置：

- `src/core/world.ts`
- `src/core/entity.ts`

完成标准：

- framework system 能通过公共 API 获取目标实体
- 常见查询不再需要直接操作 `world.entities`

当前建议语义：

- query 默认只返回当前已生效、未销毁、active 的实体
- 帧内 pending addition 不应提前出现在 query 结果里
- 帧内 pending removal 的实体因为已失活，不应继续出现在有效查询里
- `getEntitiesWithAll()` 在未传类型时返回当前 live entity snapshot
- `getEntitiesWithAny()` 在未传类型时返回空数组，避免误把“任意条件”解释成“全部实体”

---

### 3. 增加 component dependency validation

目标：

- 让错误装配尽早暴露，而不是静默失败

具体任务：

- 给 component 提供声明依赖的方式
- 在 attach 或 initialize 阶段进行检查
- 先覆盖高频依赖

优先覆盖：

- `ViewComponent -> TransformComponent`
- `ColliderComponent -> TransformComponent`
- 需要尺寸信息的能力对 `SizeComponent` 的依赖

涉及位置：

- `src/core/component.ts`
- `src/core/entity.ts`
- `src/framework/view.ts`
- `src/framework/collision.ts`

完成标准：

- 缺失关键依赖时能明确报错或至少发出高质量错误信息
- sample 中的 entity assembly 规则更清楚

当前建议实现：

- 依赖检查优先在 component initialize 阶段执行
- 错误信息需要带上 component 名、entity 名、缺失依赖名
- 第一批先覆盖 `ViewComponent` 和 `ColliderComponent`

---

### 4. 拆分 render layer

目标：

- 建立最小但清楚的渲染层次

具体任务：

- 定义类似 `backgroundLayer`、`worldLayer`、`uiLayer`、`overlayLayer`
- 让 example 不再把所有 text/sprite 都直接塞进 `renderScene.root`
- 明确 camera 将来主要作用在哪些 layer 上

涉及位置：

- `src/adapter/render-types.ts`
- `src/adapter/leafer/runtime.ts`
- `examples/dodge-blocks/dodge-blocks-scene.ts`

完成标准：

- world 对象和 UI overlay 不再混在一个裸 root 里
- 后续 camera、HUD、debug overlay 有更自然的挂载点

---

### 5. 抽出可复用状态流

目标：

- 把 sample 里的“开始 / 运行 / 暂停 / 结束”从 example 特例推进成 framework 可复用模型

具体任务：

- 提供一个轻量 `StateMachine` 或 `GameFlow`
- 把 `dodge-blocks` 中的 phase 管理迁到更通用的结构
- 明确状态切换、状态进入/退出、副作用触发的边界

涉及位置：

- `src/framework/`
- `examples/dodge-blocks/dodge-game-system.ts`

完成标准：

- 至少当前 sample 能消费抽出的状态流能力
- 下一份 sample 可以沿用同一套状态切换模型

---

## 建议执行顺序

建议按下面顺序推进：

1. 先做 `System` priority
2. 再做 `World` query API
3. 然后补 component dependency validation
4. 接着做 render layer 拆分
5. 最后抽出 reusable game flow

原因：

- 调度顺序和查询能力是 framework 复用的最底层抓手
- dependency validation 是把“能用”变成“更不容易误用”
- render layer 和 game flow 更适合在前面结构稳定后再抽

## 验收清单

发布 `v0.2.0` 前，至少确认下面这些问题答案都是“是”：

- system 顺序是否已经显式化
- 常见实体查询是否已经走公共 API
- 关键 component 缺失依赖时是否能尽早暴露
- render tree 是否已具备基础 layer 结构
- sample 的 phase 流程是否已有可复用抽象

## 与后续版本的关系

**`v0.3.0` = Engine Capability Sprint**

优先处理：

- collision 语义升级
- camera 真正落地
- prefab / factory
- asset system 正式化
- debug tooling 扩展

也就是说：

- `v0.0.2` / `0.1.x` 先把骨架站稳
- `v0.2.0` 再把 framework 做厚
- `v0.3.0` 才开始明显扩 engine-facing 能力
