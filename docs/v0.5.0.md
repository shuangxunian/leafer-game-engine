# leaferGame v0.5.0

## Version Positioning

`v0.5.0` starts the next stage:

**Data and Content Pipeline Sprint**

The previous stages made the runtime stable, made framework features reusable, added engine-facing capabilities, and introduced the first tooling layer. This stage focuses on moving content creation away from hardcoded sample code and toward data-driven structure.

---

## Why This Stage

`0.4.x` made the runtime easier to inspect.

The next bottleneck is content authoring:

- assets are still mostly registered by code
- entities are still mostly created by handwritten factory calls
- scene setup is still mostly hardcoded in sample files
- component values do not yet have schema metadata
- future editor work needs a data shape to read and write

`0.5.x` should not try to build the editor UI yet. It should build the runtime-facing data contract that an editor, loader, or content pipeline can later use.

---

## Stage Goals

`0.5.x` should move the project toward:

- resource loading instead of only resource registration
- declarative asset manifests
- entity templates/prefabs expressed as data
- scene configuration that can instantiate entities and systems
- component schema metadata for tooling and validation
- better separation between sample content and engine framework
- structures that make editor integration realistic later

---

## Suggested Patch Split

- `v0.5.1`
  - asset manifest and loading primitives
- `v0.5.2`
  - entity template data shape
- `v0.5.3`
  - scene config bootstrap
- `v0.5.4`
  - component schema and validation metadata
- `v0.5.5`
  - migrate part of `examples/dodge-blocks` to data-driven setup

---

## Current Priority

The first useful step is `v0.5.1`: asset manifest and loading primitives.

Reason:

- `AssetRegistry` already exists, but it only registers sprite metadata synchronously.
- A manifest gives content projects a stable place to declare assets.
- Loading status and missing-asset errors are needed before scene config can be reliable.
- Tooling snapshots can later expose asset loading state.

---

## Scope Boundaries

This stage should include:

- JSON-like data shapes
- runtime parsing/normalization helpers
- explicit validation errors
- tests for loading and config semantics
- small sample integration where useful

This stage should not include:

- full visual editor UI
- drag-and-drop scene editing
- tilemap editor
- remote asset pipeline
- production-grade asset bundling
- save-file/version migration system

---

## Done When

By the end of `0.5.x`:

- assets can be declared through a manifest-like data shape
- entity templates can describe reusable entity composition
- a scene can be bootstrapped from a data config at least in a minimal form
- component schema metadata exists for common framework components
- the dodge-blocks example uses at least one data-driven path
- `npm run check`, `npm test`, and `npm run build:example` pass

---

## Relation to Previous Stages

- `0.2.x` made framework reuse clearer.
- `0.3.x` made engine capabilities more concrete.
- `0.4.x` made runtime state easier to inspect.
- `0.5.x` should make content easier to describe, load, validate, and eventually edit.
