# leaferGame v0.23.1

## Version Positioning

`v0.23.1` is the first implementation pass in the `0.23.x` stage:

**Camera Viewport and Coordinate Conversion Baseline**

`v0.23.0` defined the camera runtime contract stage. This version makes `CameraSystem` expose deterministic viewport state and coordinate conversion helpers so downstream games can translate between world-space gameplay coordinates and viewport/screen-space interaction coordinates without duplicating camera math.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.23.1` does not add visual scene editors, camera timeline editors, cinematic sequencing, keyframe authoring, editor handles/gizmos, scene graph mutation UI, save/load workflows, or content production tooling.

It only hardens runtime camera math and package-facing APIs.

---

## Goals

After `v0.23.1`, the project supports:

- copied camera viewport state that includes camera center, zoom, render viewport size, world layer offset, and visible world bounds
- world-to-viewport coordinate conversion
- viewport-to-world coordinate conversion
- deterministic behavior before and after `lateUpdate(...)`
- Node-safe tests for camera math
- public docs preserving the runtime-package/not-editor boundary

---

## Scope

This version includes:

- small TypeScript data contracts for points and camera viewport state
- `CameraSystem` inspection helpers
- coordinate conversion helpers based on current camera state
- tests for zoom, offset, and round-trip conversion semantics
- public docs and package-boundary coverage

This version does not include:

- camera bounds or clamping behavior
- camera smoothing, shake, damping, or transitions
- read-only tooling snapshots
- dodge-blocks camera consumption
- visual camera controls
- editor or authoring workflows

---

## Semantics

- Camera `x` / `y` remain the world-space center point.
- Viewport coordinates use the render viewport's local pixel coordinate space.
- World-to-viewport conversion follows the same formula used to map the world layer.
- Viewport-to-world conversion is the inverse of world-to-viewport.
- The viewport snapshot must be copied so callers cannot mutate internal camera or render layer state.
- UI and overlay layers remain screen-space; this version only describes camera behavior for world-space content.
