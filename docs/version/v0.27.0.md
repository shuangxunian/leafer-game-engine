# leaferGame v0.27.0

## Version Positioning

`v0.27.0` starts the next stage:

**Pointer-First Puzzle Interaction Sprint**

`0.24.x` proved one playable 4399-style movement loop.
`0.25.x` proved a second gameplay shape can consume the engine package.
`0.26.x` extracted small framework helpers from repeated friction across those two examples.

The next stage should stop testing only keyboard/directional movement games. The product target is a Web page game engine for lightweight browser games, WeChat-mini-game-like puzzle interactions, and interactive narrative projects. The next useful proof point is a pointer-first, rule-driven puzzle loop: selecting objects, mapping pointer input to game state, updating HUD/status, and keeping puzzle rules example-owned while extracting only package-generic runtime helpers.

---

## Why This Stage

The current examples are valuable, but both are still close to arcade movement:

- `dodge-blocks`
  - movement, hazards, survival, collision, pause/resume
- `collect-stars`
  - movement, collection, timer, restart, collision

That validates real-time movement games, but not enough of the target product surface:

- tap/click-first games
- drag/drop or select/target interactions
- board/grid/pile/bottle-style puzzle state
- UI-heavy game flow
- interaction content that is closer to visual novels or lightweight narrative scenes

For games in the shape of tile-matching, sorting, pouring, stacking, or simple object-selection puzzles, the missing pressure points are no longer only movement and collision. They are pointer position, hit testing, selection state, puzzle-state transitions, small interaction feedback, and clean separation between generic input helpers and game-specific rules.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.27.0` is a planning and boundary document. It does not add public framework API by itself.

This stage may add small runtime/framework helpers later, but only when they satisfy all of these rules:

- the helper is useful for more than one pointer-first or UI-heavy game shape
- the helper is runtime code, math, input state, hit testing, scene flow, or read-only observability
- examples remain downstream consumers through package-style imports
- puzzle rules stay example-owned unless repeated examples prove a package-generic need
- no helper clones the full rules, assets, economy, progression, or UX of a specific commercial game

This stage should not add:

- visual editor UI
- scene graph editing
- puzzle/level authoring UI
- galgame script editor
- asset browser or asset manager UI
- example launcher product or game gallery
- WeChat SDK integration, platform publishing, accounts, ads, monetization, or leaderboard services
- save/load project workflow
- a generic puzzle framework before examples prove the shape

## Candidate Proof Shape

The preferred proof shape is a small pointer-first puzzle example.

Good candidates:

- a water-sort / bottle-pouring style mini puzzle
- a simple object grouping or matching puzzle
- a pile/slot selection puzzle

The example should be intentionally small. It should prove engine-package consumption, not become a content product.

The first candidate is a water-sort-style puzzle because it stresses useful engine needs without requiring physics:

- pointer/click selection
- target selection
- board/slot layout
- puzzle state transitions
- valid/invalid move feedback
- HUD/status updates
- win condition
- restart flow

Keep example-owned:

- exact puzzle rules
- level layouts
- colors/themes/assets
- move scoring
- hint rules
- undo rules
- progression
- monetization or platform integration

## Candidate Engine Gaps

### 1. Pointer Position And Click Semantics

The engine has pointer button bindings, but pointer-first games need more than "primary button is pressed".

Potential package-level shape:

- browser pointer position bridge
- current pointer position in viewport coordinates
- pressed/released/click edge state
- clear lifecycle ownership and tests

Keep out of package:

- game-specific click handlers
- UI widgets
- editor picking tools

### 2. Runtime Hit Testing Helpers

Pointer puzzle games need to map a screen/world point to an entity or region.

Potential package-level shape:

- rectangle hit testing against `TransformComponent` + `SizeComponent`
- optional layer/filter helpers
- copied hit result data for tests and tooling

Keep out of package:

- scene graph mutation UI
- inspector editing
- visual selection handles

### 3. Selection And Interaction State Patterns

Pointer-first examples need a clean place for selected source/target state.

Potential package-level shape:

- docs and example-owned pattern first
- tiny immutable state helpers only if repetition appears

Keep out of package:

- one fixed puzzle state machine
- water-sort-specific rules
- generic puzzle authoring format

### 4. UI-Heavy Flow Bridge

The stage should collect evidence for later galgame / interactive narrative work, but not implement a full visual novel framework yet.

Potential package-level shape:

- clearer usage of `GameFlow`, HUD text helpers, and screen-space UI
- notes on scene/dialogue flow needs discovered by the example

Keep out of package:

- dialogue script language
- branching story editor
- save system
- character/background asset workflow

## Done When

By the end of `0.27.x`:

- the project has a pointer-first example distinct from the two movement examples
- the example consumes engine APIs through package-style entrypoints
- pointer position, hit testing, or selection friction is either extracted into small package helpers or explicitly kept example-owned with reasons
- puzzle rules remain example-owned
- docs explain what this stage proves for Web page mini-games and what remains for fuller WeChat-mini-game-like and galgame-style projects
- no editor, launcher, gallery, marketplace, SDK integration, account system, monetization, or publishing workflow is added
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass
