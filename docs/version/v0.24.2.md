# leaferGame v0.24.2

## Version Positioning

`v0.24.2` is the second implementation pass in the `0.24.x` playable game kit stage:

**Actor Template Composition Baseline**

This version adds a small actor-facing composition helper on top of the existing `EntityTemplate` pipeline. The goal is to reduce repeated per-game boilerplate for common 2D gameplay entities such as players, enemies, hazards, pickups, and projectiles without creating an editor, visual prefab system, or scene authoring workflow.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.24.2` does not add a visual prefab editor, hierarchy editor, asset browser, drag-and-drop scene builder, component property editor, save/load project workflow, runtime mutation UI, or content publishing pipeline.

It only adds reusable runtime composition helpers that downstream games can call from code.

---

## Goals

After `v0.24.2`, the project should support:

- a Node-safe actor template data contract
- a helper that turns common actor fields into existing `EntityTemplate` component declarations
- transform, size, optional collider, and optional velocity composition
- extra component declarations for game-specific data
- dodge-blocks player template consumption through the actor helper
- tests proving copied template output and optional component behavior
- docs preserving the engine-package/not-editor boundary

---

## Scope

This version includes:

- actor template definition types in `framework`
- a `defineActorTemplate(...)` helper that returns an `EntityTemplate`
- public export and package-boundary coverage
- dodge-blocks player template integration
- Node tests for common actor component composition
- README and public API wording for the playable game kit direction

This version does not include:

- render node creation
- sprite or animation binding
- nested prefab hierarchies
- variant inheritance
- runtime prefab mutation
- visual authoring UI
- editor or content production workflows

---

## Semantics

- Actor templates are plain data and can be instantiated by the existing `instantiateEntityTemplate(...)` pipeline.
- Every actor template includes `transform` and `size`.
- Collider and velocity components are optional and included only when declared.
- Additional component declarations are copied into the returned template after built-in actor components.
- Callers should be able to mutate the original actor definition after calling `defineActorTemplate(...)` without changing the returned template.

---

## Done When

- Framework exports the actor template helper.
- Dodge-blocks uses the helper for its player template.
- Node tests cover required fields, optional collider/velocity behavior, extra component copying, and defensive output.
- Package docs explain that this is code/runtime composition, not editor prefab authoring.
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass.
