# leaferGame v0.11.0

## Version Positioning

`v0.11.0` starts the next stage:

**Sprite Animation And Asset Runtime Sprint**

The previous stage made the package boundary clearer and safer for downstream consumers. This stage returns to engine capability by extending the asset/runtime layer from loaded sprites toward reusable sprite frames, animation clips, and runtime animation playback.

---

## Why This Stage

The project already has:

- typed sprite asset registration
- async manifest loading
- browser image sprite loading
- render adapter sprite updates
- example asset preload before scene start
- package-facing import and artifact verification

The next engine bottleneck is visual content behavior.

Most 2D games need more than a single sprite fill or image source. They need named frames, animation clips, frame timing, looping rules, and a reusable system that can update render nodes without each game hand-writing animation logic.

---

## Stage Goals

`0.11.x` should move the project toward:

- sprite frame metadata
- animation clip definitions
- animation playback state
- a framework-level animation component/system pair
- tests for frame timing, looping, stopping, and missing assets
- dodge-blocks integration as a small consumer, if practical
- tooling visibility for animation state where useful

## Scope Boundaries

This stage should include:

- data contracts for frames and clips
- runtime-safe animation playback primitives
- framework integration through ECS-style components/systems
- tests for deterministic animation timing
- example usage that stays consumer-oriented

This stage should not include:

- visual editor UI
- timeline editor UI
- asset browser UI
- atlas packing tooling
- skeletal animation
- particle systems
- audio playback
- npm publishing unless explicitly decided later

---

## Done When

By the end of `0.11.x`:

- sprite animation concepts are represented by reusable engine APIs
- animations can advance deterministically in tests
- consumers can attach animation behavior to entities without hand-writing per-game timing logic
- docs explain the animation asset/runtime boundary
- examples remain consumers of the engine package boundary
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass
