# leaferGame v0.19.0

## Version Positioning

`v0.19.0` starts the next stage:

**Pointer/Input Runtime Primitives Sprint**

The previous stage closed level/map runtime primitives. This stage returns to frontend interaction foundations by extending the input layer beyond keyboard-only action bindings, starting with small Node-safe pointer button primitives before browser bridge and example integration work.

---

## Why This Stage

The project already has:

- a deterministic `InputSystem`
- keyboard bindings through `defineKeyboardBinding(...)`
- semantic action mapping through `InputActionMap`
- dodge-blocks example consumption of keyboard action mappings
- read-only tooling visibility for input actions

The next interaction bottleneck is pointer input.

Frontend games and interactive content often need pointer/mouse/touch-like intent such as select, drag, aim, or click-confirm. This should begin as reusable runtime input contracts, not as browser control panels, editor tooling, gesture authoring, or UI settings screens.

---

## Stage Goals

`0.19.x` should move the project toward:

- Node-safe pointer button bindings for `InputActionMap`
- deterministic pointer binding normalization
- browser pointer bridge integration when the runtime contract is stable
- read-only tooling formatting for pointer bindings
- example consumption where useful
- docs that keep pointer input separate from editor controls or input settings products

The `0.19.x` stage is starting.

---

## Scope Boundaries

This stage should include:

- plain TypeScript input data contracts
- deterministic validation and normalization
- Node-safe tests
- browser bridge work only after the runtime contract is clear
- read-only observability where useful

This stage should not include:

- visual input binding editor UI
- runtime rebinding screens
- local storage or user profile persistence
- gesture authoring tools
- editor controls
- browser control panels
- gamepad support unless explicitly scoped later
- npm publishing unless explicitly decided later

---

## Done When

By the end of `0.19.x`:

- downstream games can bind semantic actions to pointer buttons
- pointer bindings are validated, normalized, copied, and formatted consistently
- browser/runtime integration remains a reusable package API
- docs preserve the engine-package/not-editor boundary
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass
