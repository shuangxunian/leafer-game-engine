# leaferGame v0.30.0

## Version Positioning

`v0.30.0` starts the next stage:

**Drag/Drop And Selection Sprint**

`0.29.x` closed the responsive Web runtime baseline. The next 1.0 gap is a
stronger pointer-first interaction layer for common lightweight puzzle games:
selecting sources and targets, dragging objects, dropping onto valid regions,
and turning those interactions into game-owned actions.

This version is a planning and boundary document. It does not add public package
API by itself.

---

## Why This Stage

The engine already has:

- pointer position and pointer button runtime state
- browser pointer position bridges
- mount-local pointer coordinate helpers after resize
- rectangle hit testing and entity picking helpers
- source-target selection state helpers
- playable `pour-sort` example-owned puzzle rules
- responsive example runtime setup

The missing behavior is a reusable baseline for drag/drop style interactions
without baking any one puzzle's rules into the package. A downstream game should
be able to track an active dragged entity, resolve valid drop targets, and turn
source-target interactions into game-owned actions while keeping puzzle rules in
the example or downstream app.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.30.0` does not add a visual editor, editor selection handles, gesture
recognition system, multi-touch gameplay system, physics engine, water-sort
rules engine, undo/hint solver, visual layout designer, launcher, gallery,
marketplace, WeChat SDK wrapper, account system, ads, monetization, mobile app
shell, or publishing workflow.

This stage may add small framework helpers for selection/drag state,
source-target action resolution, and pointer puzzle example hardening. It should
not become an editor interaction layer or puzzle-specific rules framework.

---

## Candidate Engine Gaps

### 1. Selection State Hardening

The current source-target selection helper is useful, but common puzzle games
need a clearer baseline for selected entity identity, selection clearing, and
selection-derived action readiness.

Potential package-level shape:

- explicit immutable selection snapshots
- helper operations for select, clear, and replace
- source-target readiness checks that stay generic

Keep out of package:

- game-specific pouring/sorting/matching rules
- undo/hint engines
- editor selection handles

### 2. Drag State Baseline

Pointer-first games often need to carry an entity from pointer down through move
and release without rewriting the same bookkeeping.

Potential package-level shape:

- active drag identity and start/current positions
- pointer-local delta helpers
- clean cancellation and completion snapshots

Keep out of package:

- gesture recognition
- multi-touch gameplay systems
- physics simulation

### 3. Source-Target Action Baseline

After selection or drag/drop resolves a target, the package can provide a small
generic action envelope while downstream games own the rules.

Potential package-level shape:

- source id / target id action data
- validation-result shape for allowed/blocked interactions
- deterministic action snapshots for examples and tests

Keep out of package:

- water-sort, match-3, merge, or inventory rule engines
- solver, hint, scoring, or level-completion logic

### 4. Pointer Puzzle Example Hardening

The `pour-sort` example should keep proving these helpers as a downstream game,
not as engine-internal special cases.

Potential example shape:

- consume generic selection/drag/action helpers
- keep pour-sort rules in `examples/pour-sort`
- prove source-target actions still work after responsive runtime changes

Keep out of package:

- polished launcher/gallery
- visual puzzle editor
- reusable pour-sort product template

## Suggested Patch Split

- `v0.30.1`
  - selection state helper hardening
- `v0.30.2`
  - drag state baseline
- `v0.30.3`
  - source-target action baseline
- `v0.30.4`
  - pointer puzzle example hardening
- `v0.30.5`
  - stage closeout if the baseline is enough

The patch split can change if implementation reveals a smaller better slice.

## Stage Status

The `0.30.x` stage is complete through `v0.30.5`.

The closed baseline includes selection snapshots/readiness, explicit immutable
selection replacement, entity drag state snapshots, source-target action envelopes,
allowed/blocked action results, and `pour-sort` example consumption
of those generic helpers. Puzzle rules remain example-owned.

---

## Done When

By the end of `0.30.x`:

- pointer-first games can represent selected source/target state clearly
- drag state can track active entity, start/current pointer position, and
  completion/cancellation
- source-target actions have a generic package-facing shape
- `pour-sort` or another pointer puzzle example consumes the generic helpers
  while keeping game rules example-owned
- no visual editor, gesture system, multi-touch gameplay system, puzzle rules
  engine, launcher, gallery, marketplace, SDK wrapper, monetization, or
  publishing workflow is added
- `npm run check`, `npm test`, `npm run build:example`, `npm run verify:package`,
  and `npm audit --audit-level=moderate` pass
