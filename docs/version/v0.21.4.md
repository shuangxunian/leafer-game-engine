# leaferGame v0.21.4

## Version Positioning

`v0.21.4` closes the `0.21.x` stage:

**Audio Runtime Example Consumption and Boundary Closeout**

`v0.21.1` added audio contracts, `v0.21.2` added scene-owned audio runtime state, and `v0.21.3` made that state visible through read-only tooling. This version should make `examples/dodge-blocks` consume those package APIs as a downstream-style package consumer, then document the completed audio runtime boundary.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.21.4` does not add Web Audio playback, decoding, audio file loading, waveform UI, timeline editing, mixer controls, volume sliders, asset browser UI, audio authoring, project persistence, or content production workflows.

The example should record semantic audio intent and expose it through read-only tooling. It should not play sound or become an audio editor, mixer, or asset manager.

---

## Goals

After `v0.21.4`, the project should support:

- `examples/dodge-blocks` installing `AudioRuntimeSystem`
- a small example audio manifest with semantic cues/channels
- gameplay code recording deterministic audio intent for downstream-style events
- aggregate tooling opt-in for `audio: true`
- example README notes explaining audio intent and non-playback scope
- package-boundary tests proving the example consumes package APIs without editor/audio-authoring drift
- updated public docs marking `0.21.x` audio runtime primitives as closed

---

## Scope

This version should include:

- example-level audio manifest/cue ids
- example-level `addAudioRuntime(...)` consumption
- deterministic calls such as `playCue(...)` from existing gameplay events
- read-only tooling snapshot visibility through the browser panel
- README/public API/version progress updates
- tests for example consumption and boundary wording

This version should not include:

- browser audio playback
- Web Audio adapters
- audio loading or decoding
- real sound assets
- volume or mute UI
- audio editor/mixer controls
- content authoring workflows
- new gameplay scope beyond recording audio intent

---

## Semantics

- Example audio should remain optional runtime intent data.
- Audio cues should be semantic, for example `game:start`, `player:hit`, or `game:over`.
- Tooling can show audio intent records, but must not mutate runtime state.
- The example should continue using package-style imports.
- Closing the stage should document that audio remains runtime contracts/observability until a later browser playback adapter is explicitly planned.

---

## Done When

- Dodge-blocks installs audio runtime state and records deterministic cue intent.
- `createToolingSnapshot(...)` in the example opts into audio visibility.
- Example docs describe audio as runtime intent, not playback.
- Public docs mark `0.21.x` as closed after data contracts, system integration, tooling visibility, and example consumption.
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass.
