# leaferGame v0.23.3

## Version Positioning

`v0.23.3` is the third implementation pass in the `0.23.x` stage:

**Camera Read-only Tooling Snapshot Visibility**

`v0.23.1` added copied camera viewport state and coordinate conversion, and `v0.23.2` added camera bounds/follow clamping. This version should expose camera runtime state through the existing read-only tooling layer so downstream games can inspect camera center, zoom, viewport size, world layer offset, visible world bounds, and optional camera bounds without adding mutation controls.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.23.3` does not add visual scene editors, camera control panels, camera track/timeline editors, cinematic sequencing, keyframe authoring, editor handles/gizmos, scene graph mutation UI, camera authoring tools, save/load workflows, or content production tooling.

It only adds read-only runtime observability for camera state.

---

## Goals

After `v0.23.3`, the project should support:

- a Node-safe camera tooling snapshot helper
- copied camera center, zoom, viewport, visible world bounds, world layer offset, and optional bounds state
- stable text formatting for camera snapshots
- optional aggregate `createToolingSnapshot(...)` camera inclusion
- a browser tooling panel section for camera state
- Node-safe tests for snapshot copying and formatting
- public docs preserving the runtime-package/not-editor boundary

---

## Scope

This version includes:

- read-only camera snapshot data contracts
- `tooling` helpers for camera state
- aggregate tooling opt-in where useful
- browser panel section formatting
- tests for missing camera system behavior and copied snapshot output
- public docs and package-boundary coverage

This version does not include:

- camera mutation controls
- pan/zoom buttons
- drag-to-move camera UI
- camera target selection UI
- visual camera handles/gizmos
- timeline/keyframe authoring
- dodge-blocks camera consumption
- editor or authoring workflows

---

## Semantics

- Camera tooling reads `CameraSystem`; it does not create, move, zoom, follow, clamp, or destroy cameras.
- Snapshot output should be copied and stable so callers cannot mutate camera internals.
- Aggregate tooling inclusion should be opt-in, matching existing `audio`, `collisions`, and `inputActions` style.
- Browser panel output should be informational only and should not add buttons, sliders, handles, or editing controls.
- Missing camera systems should produce a safe empty/missing snapshot instead of throwing.

---

## Done When

- Camera state can be captured through Node-safe tooling helpers.
- Aggregate tooling can include camera state through an explicit opt-in.
- Browser tooling can render a read-only camera section.
- Package-boundary tests discover the version docs and public API wording.
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass.
