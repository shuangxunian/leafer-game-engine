# leaferGame v0.10.0

## Version Positioning

`v0.10.0` starts the next stage:

**Public API And Package Boundary Sprint**

The previous stage made gameplay flow and scene startup lifecycle reusable. This stage focuses on making `leaferGame` feel more like a real frontend engine dependency package: clear imports, stable package boundaries, package-level smoke tests, and docs that teach consumers which entrypoint to use.

---

## Why This Stage

The project now has useful engine capabilities across `core`, `framework`, `runtime`, `adapter`, and `tooling`.

For a package consumer, the next risk is not missing gameplay features. The next risk is uncertainty:

- Which APIs are public?
- Should users import from the root package or subpaths?
- Which browser-only APIs pull in `Leafer`?
- Can Node-side tests import non-browser modules safely?
- Does `npm pack` contain the files expected by downstream users?
- Are examples using the same package boundaries that real consumers would use?

This sprint should reduce those uncertainties before adding another large gameplay or rendering feature area.

---

## Stage Goals

`0.10.x` should move the project toward:

- explicit public API entrypoints
- package import smoke tests
- safer separation between browser runtime APIs and Node-safe engine APIs
- clearer README import guidance
- package artifact verification through `npm pack`
- example imports that resemble downstream usage where practical
- documentation that treats this repository as an engine package, not an editor

## Scope Boundaries

This stage should include:

- package-facing API documentation
- root export and subpath export verification
- Node-safe import tests for non-browser entrypoints
- browser-runtime import boundary documentation
- package artifact checks
- example cleanup that improves dependency-package realism

This stage should not include:

- visual editor UI
- asset browser UI
- scene authoring tools
- new gameplay scope
- large rendering feature work
- publishing a new npm version unless explicitly decided later

---

## Done When

By the end of `0.10.x`:

- package entrypoints are documented and smoke-tested
- consumers can understand when to use root imports vs subpath imports
- Node tests can verify public non-browser APIs without accidentally importing browser-only runtime code
- package artifacts can be checked before publish
- examples remain consumers of the engine package boundary
- `npm run check`, `npm test`, `npm run build:example`, and the package verification command pass
