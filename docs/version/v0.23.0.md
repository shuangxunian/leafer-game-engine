# leaferGame v0.23.0

## Version Positioning

`v0.23.0` starts the next stage:

**Camera Runtime Contract Hardening Sprint**

The engine already has a minimal `CameraSystem` that can move, zoom, follow an entity, and drive the world render layer. The next step should turn camera behavior into a clearer package-facing runtime contract that downstream games can rely on across gameplay code, rendering adapters, tests, examples, and read-only tooling.

---

## Why This Stage

Camera is a core 2D engine primitive, but the current implementation is intentionally small:

- camera position and zoom can drive the world layer
- follow behavior can track a target transform
- UI and overlay layers stay screen-space through the render layer contract

The missing package-facing pieces are:

- reusable viewport/world coordinate conversion helpers
- explicit camera viewport state that downstream games can inspect
- optional bounds/clamping semantics for follow and manual camera movement
- read-only tooling visibility for camera state
- example consumption that proves camera APIs remain runtime/game-facing
- public docs that keep camera separate from scene editing, camera authoring tools, or visual timeline products

This remains frontend 2D game engine package work, not an editor, cinematic timeline, scene graph authoring surface, camera track editor, or content production workflow.

---

## Stage Goals

`0.23.x` should move the project toward:

- Node-safe camera math and viewport data contracts
- clearer `CameraSystem` state inspection APIs
- deterministic world/screen coordinate conversion behavior
- optional camera bounds/clamping primitives
- read-only camera tooling snapshots where useful
- downstream-style example consumption without demo-specific camera hacks
- package-facing documentation for camera runtime boundaries

---

## Scope Boundaries

This stage should include:

- plain TypeScript camera data contracts
- deterministic camera math helpers
- `CameraSystem` runtime behavior hardening
- Node-safe tests
- read-only runtime observability
- example consumption where useful
- docs that frame camera as engine runtime behavior

This stage should not include:

- visual scene editors
- camera track/timeline editors
- cinematic sequencing tools
- keyframe authoring
- editor handles or gizmos
- scene graph mutation UI
- content save/load or project-file workflows
- adapter-specific camera products beyond the existing render layer contract

---

## Done When

By the end of `0.23.x`:

- downstream games can convert between world and viewport coordinates through reusable camera APIs
- camera bounds/follow behavior is deterministic and tested
- camera state can be observed through read-only tooling without mutation controls
- dodge-blocks consumes camera runtime APIs only if it clarifies package behavior
- public docs preserve the runtime-package/not-editor boundary
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass
