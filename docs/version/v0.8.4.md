# leaferGame v0.8.4

## Version Positioning

`v0.8.4` is the fourth `0.8.x` delivery pass:

**Asset Load State Snapshot And Tooling Visibility**

This version makes the new asset loading state visible through the existing runtime debug and tooling snapshot surfaces.

---

## Product Boundary

This remains engine package work.

`v0.8.4` exposes read-only asset state for developers and upper-layer tools. It does not add an editor, asset browser, import workflow, or visual authoring UI inside this repository.

---

## Goals

After `v0.8.4`, the project should support:

- debug snapshots that include sprite load state
- readable asset state formatting
- a dedicated tooling panel section for assets
- clear loaded/failed/loading/registered counts
- stable per-sprite error visibility for failed loads

---

## Scope

This version includes:

- richer `DebugAssetSnapshot` data
- copied per-sprite load state in snapshots
- `formatDebugAssetSnapshot(...)`
- `createAssetsPanelSection(...)`
- tests for snapshot shape, formatting, and panel section composition

This version does not include:

- editor asset browser UI
- asset mutation from tooling
- retry controls
- progress percentages
- network waterfall views
- cache management
- example integration cleanup

---

## Semantics

- Asset tooling is read-only.
- Snapshot data should not expose mutable registry internals.
- Registered sprites without a finished load should appear as `registered`.
- Failed sprites should expose their error message.
- Runtime debug output should stay compact, while the dedicated assets section can show per-sprite rows.

---

## Done When

- Asset load state appears in debug/tooling snapshots.
- Tooling panel sections include a dedicated asset section when asset data exists.
- Existing runtime inspector behavior remains compatible.
- `npm run check`, `npm test`, and `npm run build:example` pass.
