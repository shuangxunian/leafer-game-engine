# leaferGame v0.15.2

## Version Positioning

`v0.15.2` is the second implementation pass in the `0.15.x` stage:

**Safe Scene Bootstrap Validation Gate**

This version connects the scene config validation baseline to bootstrap as an opt-in safety gate so downstream games can prevent partial runtime mutation when config diagnostics are already known.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.15.2` does not add editor UI, scene graph editing, component editing, asset management UI, runtime mutation controls, content publishing, remote content pipelines, or npm publishing. It only adds safer package-level bootstrap semantics for data-driven scene consumption.

---

## Goals

After `v0.15.2`, the project should support:

- an opt-in `validateBeforeBootstrap` bootstrap option
- bootstrap results that can include validation diagnostics
- no scene/entity/system/asset mutation when opt-in validation fails
- component data validation diagnostics for entity template factories
- tests proving safe bootstrap failure returns diagnostics instead of throwing for expected invalid config data
- public docs explaining that bootstrap validation is an engine package safety feature, not an editor workflow

---

## Scope

This version includes:

- `SceneBootstrapOptions.validateBeforeBootstrap`
- `SceneBootstrapResult.validation`
- validation diagnostics for invalid entity template component data
- bootstrap short-circuit behavior when opt-in validation fails
- Node tests for safe bootstrap validation failure and successful validated bootstrap
- README/public API progress updates

This version does not include:

- making validation mandatory for all bootstrap calls
- async validation
- asset file existence checks
- save/load systems
- editor controls
- visual content authoring
- example gameplay changes

---

## Semantics

- Default `bootstrapSceneFromConfig(...)` behavior remains compatible.
- When `validateBeforeBootstrap` is true, bootstrap runs `validateSceneConfig(...)` before mutating runtime state.
- If validation fails, bootstrap returns empty `entities` and `systems`, does not register assets, and includes the validation result.
- If validation passes, bootstrap proceeds normally and includes the successful validation result.
- Expected invalid config data should be represented as diagnostics instead of uncaught validation-time exceptions.

---

## Done When

- Safe bootstrap validation is opt-in and package-facing.
- Invalid configs can stop bootstrap before scene/entity/system/asset mutation.
- Component data errors can be inspected through validation diagnostics.
- Existing bootstrap behavior remains covered.
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass.
