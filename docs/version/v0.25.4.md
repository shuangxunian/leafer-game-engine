# leaferGame v0.25.4

## Version Positioning

`v0.25.4` closes the `0.25.x` second playable example stage:

**Second Playable Example Docs And Stage Closeout**

`0.25.x` started as a proof question: can the engine package support a second browser mini-game shape outside `examples/dodge-blocks` without drifting into an editor, launcher, gallery, or one-off demo pile?

The answer is now yes at the baseline level. `examples/collect-stars` is routed, build-covered, playable, documented, package-facing, and organized as a small downstream-style game consumer.

No new framework API is added in this closeout.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.25.4` does not add a visual editor, example launcher product, game gallery, scene builder, asset manager, level authoring workflow, project save/load workflow, online service, account system, leaderboard, monetization, or publishing surface.

It closes documentation and package-boundary coverage for the second playable example stage.

---

## Stage Outcome

After `0.25.x`, the repository has two downstream-style playable examples:

- `examples/dodge-blocks`
  - falling-hazard survival loop
  - resource loading, sprite animation, audio runtime/playback, level config, tooling panel, pointer input, and gameplay snapshot coverage
- `examples/collect-stars`
  - bounded collection/timer loop
  - semantic keyboard actions, normalized movement, actor templates, HUD, tile-backed playfield visuals, collision collection, and package-facing local gameplay modules

Together they prove the existing package APIs are not only dodge-blocks-specific:

- browser runtime and scene lifecycle can boot multiple examples
- examples can import through package-style entrypoints
- input actions can describe different gameplay controls
- movement helpers can support bounded arcade movement
- actor templates can create reusable ECS data for different actors
- HUD helpers can support score/status/timer text
- tile map layer views can create world/background visuals
- collision queries can drive different gameplay outcomes
- `GameFlow` can represent distinct phase models
- example code can stay downstream-owned without becoming engine internals

---

## What We Did Not Extract

`collect-stars` intentionally did not force new public framework helpers yet.

The stage found that the current APIs are enough for a second baseline loop when the example is organized well. Future extraction should wait for repeated friction across more examples, especially around:

- actor factory convenience
- common bounded-player movement components
- reusable score/timer/game-loop helpers
- reusable random spawn helpers
- optional observability hooks for example-owned gameplay systems

These should remain runtime/framework/package APIs if they are added later. They should not become editor, authoring, launcher, template marketplace, or content publishing features in this repository.
