# leaferGame v0.28.5

## Version Positioning

`v0.28.5` closes the `0.28.x` real sprite / image rendering stage:

**Real Sprite / Image Rendering Stage Closeout**

`0.28.x` started because sprite asset `source` data existed, but examples and
render nodes still leaned too heavily on colored rectangle placeholders. The
stage is now complete at the baseline level needed before `1.0.0`.

No new public package API is added in this closeout.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.28.5` does not add an asset editor, visual asset manager, atlas packer,
sprite editor, image cropper, asset browser UI, CDN pipeline, bundled art
library, marketplace, launcher, gallery, WeChat SDK wrapper, account system,
ads, monetization, or publishing workflow.

It closes documentation and package-boundary coverage for the real sprite /
image rendering stage.

---

## Stage Outcome

After `0.28.x`, the engine package has a source-backed sprite rendering
baseline:

- image-backed Leafer sprite adapter
  - `RenderSpriteAsset.source` reaches the native Leafer `Image.url`.
  - fill-only sprite assets keep the existing placeholder rectangle behavior.
- asset registry to render-node handoff
  - `AssetRegistry.getSpriteRenderAsset(id)` and
    `AssetRegistry.requireSpriteRenderAsset(id)` return copied render asset
    data without leaking registry-only state.
  - `SpriteAnimationSystem` applies registered render assets to
    sprite-capable view nodes.
  - `attachActorSpriteView(...)` can resolve a sprite through `assets + assetId`.
- example image-like consumption
  - `examples/dodge-blocks` consumes source-backed sprite assets for player and
    hazard views through package-facing APIs.
- package boundary documentation
  - `docs/sprite-rendering.md` explains the runtime rendering boundary,
    `/framework` DOM-free constraints, example-owned art data, and asset
    authoring exclusions.

Together these pieces let downstream browser games declare image-like sprite
sources for simple characters, items, backgrounds, icons, and puzzle objects
without turning the engine package into an asset production suite.

---

## What Stayed Out Of Scope

The following remains intentionally outside the package:

- visual asset management
- sprite/atlas/image authoring
- image cropping and editing
- CDN or production asset pipeline automation
- bundled commercial art packs
- asset marketplace, launcher, gallery, or template store
- WeChat SDK wrapper, publishing workflow, accounts, ads, or monetization

These are possible downstream products or application concerns. The package
only owns the runtime/render contract, framework handoff, adapter behavior,
package docs, tests, and example consumption proof.

---

## Remaining Gaps Toward 1.0

This stage closes real sprite / image rendering at the baseline level, but it
does not finish the full `1.0.0` target.

Important remaining areas:

- responsive Web runtime behavior across container sizes and mobile-ish
  viewports
- pointer coordinate conversion after resize
- drag/drop state for object-moving puzzle interactions
- screen-space UI, dialogue, choices, and scene flow for galgame-style
  prototypes
- quick-start game kit helpers that reduce boilerplate for new small games

Those should be handled by later stages instead of stretching `0.28.x` into an
asset manager or content production workflow.
