# leaferGame v0.12.0

## Version Positioning

`v0.12.0` starts the next stage:

**Runtime Services And Event Pipeline Sprint**

The previous stage made sprite animation reusable across asset metadata, deterministic playback, ECS behavior, example consumption, and read-only tooling visibility.

This stage moves back to engine runtime foundations by adding small, Node-safe services that can decouple gameplay systems without turning the repository into an editor or app framework.

---

## Why This Stage

The project already has:

- stable ECS-style scene/entity/component/system primitives
- package-facing subpath exports
- resource loading and sprite animation runtime foundations
- GameFlow and scene lifecycle helpers
- read-only runtime tooling snapshots

The next bottleneck is cross-system communication.

Games need a reusable way for systems to publish and react to gameplay/runtime events without hard-coding references between every system. Later engine features such as audio triggers, save checkpoints, scene transitions, task scheduling, analytics hooks, and upper-layer tooling adapters can all consume the same event pipeline.

---

## Stage Goals

`0.12.x` should move the project toward:

- a deterministic, Node-safe event bus
- clear subscription, one-shot subscription, unsubscribe, and cleanup semantics
- stable event envelopes that include type, payload, and sequence metadata
- tests for event ordering, mutation during dispatch, and listener cleanup
- eventual runtime scheduling/timer helpers built on deterministic time updates
- docs that frame these services as engine package APIs, not editor infrastructure

## Current Progress

The `0.12.x` stage is closed at `v0.12.5`.

- `v0.12.1` added a deterministic, Node-safe framework `EventBus` with subscription, one-shot listener, unsubscribe, clear, listener count, event envelope, and dispatch mutation semantics.
- `v0.12.2` added a deterministic, Node-safe `RuntimeScheduler` for one-shot and repeated update-driven tasks.
- `v0.12.3` added opt-in scene integration for runtime services through `RuntimeServicesSystem`.
- `v0.12.4` added read-only tooling snapshots and panel sections for runtime services state.
- `v0.12.5` documented the runtime services boundary and aligned package-facing docs with the completed sprint.

---

## Scope Boundaries

This stage should include:

- pure TypeScript framework/runtime services
- deterministic behavior that can be tested in Node
- package-safe exports from `/framework`
- documentation for downstream game consumption

This stage should not include:

- visual editor UI
- editor message bus or project graph infrastructure
- asset browser UI
- DOM-dependent runtime panels as the primary feature
- network transport
- analytics SDK integration
- npm publishing unless explicitly decided later

---

## Done When

By the end of `0.12.x`:

- downstream games can use engine-provided runtime services instead of hand-rolled per-game helpers
- event dispatch order and listener lifecycle behavior are deterministic
- timer/scheduler behavior can be advanced in tests without a browser loop
- public docs explain the runtime service boundary
- examples remain consumers of the engine package boundary
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass
