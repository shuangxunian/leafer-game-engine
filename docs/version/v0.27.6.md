# leaferGame v0.27.6

## Version Positioning

`v0.27.6` closes the `0.27.x` pointer-first puzzle interaction stage:

**Pointer-First Puzzle Interaction Stage Closeout**

`0.27.x` started as a pressure test for browser mini-games that are not primarily keyboard movement games. The stage is now complete at the baseline level.

No new public package API is added in this closeout.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.27.6` does not add a visual editor, puzzle authoring UI, galgame script editor, scene graph editing, asset browser, example launcher, game gallery, marketplace, WeChat SDK wrapper, account system, ads, monetization, leaderboard, analytics service, save/load project workflow, or publishing workflow.

It closes documentation and package-boundary coverage for the pointer-first puzzle interaction stage.

---

## Stage Outcome

After `0.27.x`, the engine package has three small pointer-first runtime/framework helpers:

- pointer position runtime state
  - `InputSystem.setPointerPosition(...)`, `InputSystem.getPointerPosition()`, and `InputSystem.clearPointerPosition()` let downstream games observe the current pointer position in a Node-safe input system.
  - `BrowserPointerPositionBridge` writes browser pointer coordinates into that state.
- rectangle hit testing and entity picking
  - `pointInRect(...)`, `getEntityHitRect(...)`, `hitTestEntitiesAtPoint(...)`, and `pickTopEntityAtPoint(...)` map pointer-like coordinates to runtime entities through ECS component data.
- source-target selection state
  - `createSourceTargetSelectionState(...)`, `selectSourceTargetSource(...)`, `selectSourceTargetTarget(...)`, `clearSourceTargetTarget(...)`, `clearSourceTargetSelection(...)`, and `getSourceTargetSelectionPair(...)` provide a tiny immutable state pattern for "pick one thing, then pick a target" interactions.

Together these helpers support click/tap-first games such as sorting, grouping, matching, pouring, and simple point-and-click puzzle flows without turning the package into a puzzle rules framework.

---

## Example Proof

`examples/pour-sort` now stands beside `examples/dodge-blocks` and `examples/collect-stars` as a third downstream-style playable example.

It proves:

- browser route and build coverage for a pointer-first example
- pointer position and pointer button bridges working together
- entity picking through package-facing hit testing helpers
- source-target selection state consumed by a downstream-style scene
- HUD/status feedback for moves, invalid moves, and solved state
- example-owned bottle state, top-color pour validation, liquid rendering, and read-only gameplay snapshot

The exact puzzle rules stay in `examples/pour-sort`. They are not exported from `/framework` and should not become a water-sort framework unless more downstream examples prove a package-generic need.

---

## What Stayed Example-Owned

The following remains intentionally outside `/framework`:

- bottle capacity and level layout
- color stack representation
- top-color move validation
- invalid move copy
- solved-state rules
- liquid segment rendering choices
- move counting semantics
- undo, hints, scoring, progression, and restart policy
- gameplay snapshot fields

These are game-specific decisions. The engine package supplies runtime primitives and small reusable interaction helpers.

---

## Remaining Gaps Toward 1.0

This stage improved pointer-first puzzle support, but it does not finish the full `1.0.0` target.

Important remaining areas:

- real image-backed sprite rendering and asset handoff
- responsive resize and pointer coordinate behavior after resize
- drag/drop state for object-moving puzzle interactions
- screen-space UI and scene flow hardening
- dialogue and choice-flow baseline for galgame / interactive narrative prototypes
- quick-start game kit helpers that reduce boilerplate for new small games

Those should be handled by later stages instead of stretching `0.27.x` into an editor or content platform.
