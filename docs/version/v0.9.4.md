# leaferGame v0.9.4

## Version Positioning

`v0.9.4` is the fourth `0.9.x` delivery pass:

**Scene Lifecycle Start Helper**

This version adds a small runtime-level helper for the common flow of preparing a scene asynchronously before starting it.

---

## Product Boundary

This remains engine package runtime work.

`v0.9.4` does not add loading-screen UI, app routing, scene editor flows, or visual authoring behavior. It only provides a reusable lifecycle helper that examples and downstream games can call.

The helper belongs to the runtime/package boundary. It should not introduce editor concepts into this repository; future editor-like products can consume the lifecycle state from a separate upper-layer project if needed.

---

## Goals

After `v0.9.4`, the project should support:

- explicit scene start phases for idle/loading/ready/running/failed
- async preparation before `runtime.start(scene)`
- deterministic success and failure results
- phase transition callbacks for logs or tooling
- dodge-blocks using the helper for asset preload before scene start

---

## Scope

This version includes:

- `SceneLifecyclePhase`
- `SceneLifecycleTransition`
- `startSceneWithLifecycle(...)`
- tests for success, failure, no-prepare, and transition ordering
- dodge-blocks integration as a consumer

This version does not include:

- loading screen widgets
- progress bars
- app router integration
- scene editor UI
- save/load orchestration
- multi-scene transition effects

---

## Semantics

- The helper starts from `idle`.
- The helper enters `loading` before optional preparation.
- After preparation succeeds, or when no preparation is provided, it enters `ready`.
- It calls the supplied start function only after readiness.
- If start succeeds, it enters `running`.
- If preparation or start fails, it enters `failed` and returns a failed result.

---

## Done When

- The runtime package exports a scene lifecycle helper.
- Dodge-blocks uses the helper instead of manually preloading before `runtime.start(...)`.
- Existing scene and runtime behavior remain compatible.
- `npm run check`, `npm test`, and `npm run build:example` pass.
