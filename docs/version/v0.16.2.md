# leaferGame v0.16.2

## Version Positioning

`v0.16.2` is the second implementation pass in the `0.16.x` stage:

**Sprite-Capable Render Node Contract**

This version makes the existing sprite-capable render-node expectation explicit and reusable. Sprite animation already requires a `ViewComponent` render node with `setAsset(...)`; this patch turns that requirement into a package-facing framework helper instead of keeping it as private animation-system duck typing.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.16.2` does not add editor UI, visual inspectors, scene graph editing, component editing, asset browser UI, runtime mutation controls, sprite authoring tools, atlas packing, or content production workflows. It only clarifies a reusable render-node capability contract that downstream games and engine systems can test.

---

## Goals

After `v0.16.2`, the project should support:

- a public `SpriteCapableRenderNode` framework type
- a public `isSpriteCapableRenderNode(...)` capability guard
- `SpriteAnimationSystem` using the shared guard instead of a private duplicate
- tests proving sprite-capable and non-sprite render-node detection
- README/public API updates that keep the feature framed as engine package work

---

## Scope

This version includes:

- shared render-node capability typing
- a Node-safe capability guard
- sprite animation system cleanup
- focused tests for the guard and existing error path
- docs and roadmap links

This version does not include:

- render adapter redesign
- new rendering features
- sprite atlas/source-rect rendering
- render-node pooling
- example gameplay changes
- editor controls
- asset management UI

---

## Semantics

- A sprite-capable render node is any `RenderNode`-like object that exposes `setAsset(asset)`.
- The capability guard is intentionally structural so adapter implementations can remain decoupled.
- `SpriteAnimationSystem` still applies frame sprite assets only when an entity has a `ViewComponent` whose node is sprite-capable.
- Non-sprite nodes still fail clearly when animation tries to apply a sprite frame to them.

---

## Done When

- The capability guard is exported from the Node-safe `framework` entrypoint.
- Sprite animation uses the shared guard.
- Existing sprite animation behavior remains compatible.
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass.
