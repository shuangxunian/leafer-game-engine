# leaferGame v0.16.0

## Version Positioning

`v0.16.0` starts the next stage:

**Render/View Contract Hardening Sprint**

The previous stage closed data-driven scene config contracts. This stage returns to render-facing engine fundamentals: how ECS entities, render nodes, render layers, camera mapping, sprite-capable nodes, and example code should interact.

---

## Why This Stage

The project already has:

- `RenderAdapter`, `RenderScene`, `RenderNode`, and typed render layers
- `ViewComponent` binding ECS entities to render nodes
- `CameraSystem` driving the world layer
- sprite animation applying frame assets to sprite-capable `ViewComponent` nodes
- `examples/dodge-blocks` using world/ui/overlay layers explicitly
- tooling snapshots that can observe render viewport/layer state

The next render-facing bottleneck is explicit contract clarity.

Downstream games need to know which render behavior is guaranteed by the engine package, which behavior belongs to an adapter, and which behavior remains game-specific wiring. This should remain engine package work, not an editor, scene graph authoring UI, or visual node inspector.

---

## Stage Goals

`0.16.x` should move the project toward:

- clearer `ViewComponent` sync semantics
- reusable helpers for render-node capability checks where useful
- better docs for render adapter, render scene, and layer ownership
- stronger tests around view synchronization and render-node lifecycle
- example coverage where useful

---

## Scope Boundaries

This stage should include:

- render-node and view synchronization semantics
- adapter-facing contract docs
- Node-safe tests for pure framework behavior
- example/build verification for browser-facing adapter behavior

This stage should not include:

- visual scene editor UI
- scene graph editing
- component value editing UI
- asset browser UI
- render timeline tooling
- shader/material authoring systems
- npm publishing unless explicitly decided later

---

## Done When

By the end of `0.16.x`:

- downstream games can understand how ECS view components map to render nodes
- render layer responsibilities are documented
- sprite-capable render-node expectations are explicit
- examples remain consumers of the engine package boundary
- docs preserve the engine-package/not-editor boundary
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass
