# leaferGame v0.26.2

## Version Positioning

`v0.26.2` continues the `0.26.x` framework extraction stage:

**Gameplay Snapshot Convention Baseline**

`examples/dodge-blocks` already exposes a small read-only gameplay snapshot from its example-owned game system. `examples/collect-stars` has the same kind of example-owned gameplay state: phase, score, remaining time, active star state, and whether gameplay is currently running.

This version aligns the second playable example with that convention.

It adds no new public package API.

It does not add a generic framework gameplay snapshot API. The snapshot remains owned by the example system because score, timer, star count, pause behavior, best score, and win/loss rules are game-specific.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.26.2` documents and applies an example-owned read-only gameplay snapshot pattern. It does not add a visual editor, gameplay debugger UI, mutable inspector, scene builder, prefab editor, launcher, gallery, marketplace, save/load workflow, leaderboard, account system, analytics service, or publishing workflow.

The engine package can provide runtime primitives and read-only observability conventions. It should not absorb game-specific scoring, timers, spawn rules, or progression logic into `/framework`.

---

## Goals

After `v0.26.2`, both playable examples should:

- keep gameplay rules inside the example-owned game system
- expose a small `getGameplaySnapshot()` method for read-only inspection
- include current `GameFlow` phase in that snapshot
- expose enough state to confirm the playable loop is alive
- avoid mutation APIs from the snapshot path
- avoid adding a generic gameplay debugger or framework-level scoring/timer abstraction

---

## Scope

This version includes:

- `CollectStarsGameplaySnapshot` in `examples/collect-stars`
- `CollectStarsGameSystem.getGameplaySnapshot()`
- `CollectStarsScene.getGameplaySnapshot()` as a scene-level read-only forwarding convenience
- collect-stars boot logging the gameplay snapshot after lifecycle start
- docs and package-boundary tests covering the convention
- a small cleanup so remaining bounded reset logic can use `clampPositionToBounds(...)`

This version does not include:

- a public `/framework` gameplay snapshot helper
- a generic score system
- a generic timer system
- a generic collect/dodge gameplay loop
- runtime mutation controls
- editor or launcher UI

---

## Semantics

- Snapshot objects are plain data returned by example-owned game systems.
- Snapshot fields should describe current runtime state, not provide mutation handles.
- The snapshot may include game-specific fields when those fields are only used for read-only inspection.
- The snapshot convention can be documented across examples without becoming a framework API.

---

## Done When

- `collect-stars` exposes a read-only gameplay snapshot.
- `dodge-blocks` and `collect-stars` both keep gameplay snapshot logic example-owned.
- Package-boundary tests prove no editor, launcher, mutable inspector, or generic gameplay framework is introduced.
- README, public API inventory, roadmap, and example docs describe the new boundary.
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass.
