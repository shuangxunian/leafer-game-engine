# leaferGame v0.24.1

## Version Positioning

`v0.24.1` is the first implementation pass in the `0.24.x` playable game kit stage:

**Normalized Directional Movement Baseline**

This version extracts a small but important arcade movement primitive into the framework layer. Downstream games often read directional actions such as left/right/up/down and turn them into an `(x, y)` movement vector. Without normalization, diagonal movement becomes faster than horizontal or vertical movement.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.24.1` does not add player controllers for a specific game, visual control editors, keybinding UI, character editors, physics simulation, animation state machines, or content authoring workflows.

It only adds reusable runtime math for gameplay movement and updates the example to consume it.

---

## Goals

After `v0.24.1`, the project should support:

- a Node-safe movement vector data contract
- a helper that caps directional movement magnitude so diagonals are not faster
- stable behavior for zero vectors, axis-aligned movement, diagonal movement, and analog-like partial vectors
- framework public exports for the movement helper
- dodge-blocks player movement consuming the framework helper
- tests proving the movement helper behavior
- docs preserving the engine-package/not-editor boundary

---

## Scope

This version includes:

- movement vector helper API in `framework`
- unit tests for movement vector clamping
- dodge-blocks player movement integration
- README and package-boundary updates pointing `0.24.x` toward playable game primitives

This version does not include:

- a full character controller
- acceleration, friction, knockback, slopes, or platformer physics
- collision response
- pathfinding
- gamepad analog input adapters
- visual input binding UI
- editor or authoring workflows

---

## Semantics

- A zero movement vector remains zero.
- A vector with length less than or equal to the configured maximum keeps its magnitude.
- A vector longer than the configured maximum is scaled down while preserving direction.
- The default maximum length is `1`, which is appropriate for digital directional input.
- Invalid or negative maximum lengths should fail clearly instead of silently producing broken movement.
