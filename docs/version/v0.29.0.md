# leaferGame v0.29.0

## Version Positioning

`v0.29.0` starts the next stage:

**Responsive Web Runtime Sprint**

`0.28.x` closed the real sprite / image rendering baseline. The next 1.0 gap is
runtime behavior across browser container sizes: examples currently assume a
fairly fixed viewport shape, and pointer-first games need pointer coordinates to
remain meaningful after resize.

This version is a planning and boundary document. It does not add public package
API by itself.

The `0.29.x` stage is complete through `v0.29.5`; see
[`v0.29.5.md`](v0.29.5.md) for the responsive Web runtime closeout.

---

## Why This Stage

The engine already has:

- browser runtime assembly through `createBrowserRuntime(...)`
- render scene viewport metadata
- camera viewport/world coordinate conversion helpers
- browser pointer position and pointer button bridges
- pointer-first entity picking and source-target selection helpers
- playable examples that should remain usable in desktop and mobile-ish
  viewport shapes

The missing behavior is a clear runtime contract for resizing. A downstream Web
game should be able to mount into a container, react to size changes, keep
render viewport metadata in sync, and map pointer coordinates consistently after
the viewport changes.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.29.0` does not add a responsive page builder, device compatibility matrix,
mobile app shell, WeChat SDK wrapper, publishing workflow, layout editor, visual
UI designer, launcher, gallery, marketplace, account system, ads, or
monetization.

This stage may add adapter/runtime contracts, browser resize bridges, pointer
coordinate conversion helpers, and example verification. It should not become a
platform wrapper or app shell.

---

## Candidate Engine Gaps

### 1. Render Scene Resize Contract

Render scenes already expose viewport size, but downstream code needs a stable
way to update and inspect that state when a container changes.

Potential package-level shape:

- explicit viewport width/height update behavior
- copied viewport state for tests and tooling
- clear rules for render layers and camera state after resize

Keep out of package:

- visual layout editor
- CSS framework or page shell
- per-device compatibility database

### 2. Browser Resize Bridge

Browser games need an opt-in bridge that reads mount/container size and updates
the runtime without forcing a specific app layout.

Potential package-level shape:

- observe a mount element or window-like target
- update render scene dimensions deterministically
- detach cleanly during runtime teardown

Keep out of package:

- project template generator
- mobile app wrapper
- platform publishing SDK integration

### 3. Pointer Coordinates After Resize

Pointer-first puzzle examples need coordinates that remain consistent after the
render viewport changes.

Potential package-level shape:

- document viewport/local coordinate expectations
- make browser pointer position bridge account for mount bounds where needed
- prove entity picking still works after resize in tests or examples

Keep out of package:

- complex gesture recognition
- multi-touch gameplay system
- editor selection handles

### 4. Example Responsive Verification

At least one playable example should prove that the runtime remains usable in a
desktop and mobile-ish viewport.

Potential example shape:

- use a smaller/narrower viewport in tests or example docs
- verify HUD, playfield, pointer position, and render scene state do not assume
  one fixed size
- keep game-specific responsive layout choices example-owned

Keep out of package:

- polished responsive landing page
- example launcher product
- design system or UI builder

## Suggested Patch Split

- `v0.29.1`
  - render scene resize contract
- `v0.29.2`
  - browser resize bridge
- `v0.29.3`
  - pointer coordinate behavior after resize
- `v0.29.4`
  - example responsive verification
- `v0.29.5`
  - stage closeout if the evidence says the baseline is enough

The patch split can change if implementation reveals a smaller better slice.
The stage is closed through `v0.29.5`.

---

## Done When

By the end of `0.29.x`:

- runtime can react to container resize
- render viewport width/height can update safely
- pointer coordinates can be mapped consistently after resize
- DPR / high-density display behavior is documented or minimally handled
- examples remain usable on desktop and mobile-ish viewports
- no visual editor, page builder, launcher, gallery, marketplace, SDK wrapper,
  mobile app shell, monetization, or publishing workflow is added
- `npm run check`, `npm test`, `npm run build:example`, `npm run verify:package`,
  and `npm audit --audit-level=moderate` pass
