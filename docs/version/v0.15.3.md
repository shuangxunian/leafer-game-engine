# leaferGame v0.15.3

## Version Positioning

`v0.15.3` is the third implementation pass in the `0.15.x` stage:

**Dodge Blocks Scene Config Consumption Polish**

This version proves the data-driven scene contract through the existing browser example without turning the example into a visual editor or authoring workflow.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.15.3` does not add editor UI, scene graph editing, component editing, asset management UI, runtime mutation controls, content publishing, remote content pipelines, or npm publishing. It only makes the example consume reusable scene config and safe bootstrap APIs as a downstream game would.

---

## Goals

After `v0.15.3`, the project should support:

- a `dodge-blocks` scene config factory for static example content
- asset manifest and player base entity data declared through one scene config shape
- safe player entity bootstrap through `bootstrapSceneFromConfig(..., { validateBeforeBootstrap: true })`
- validation errors that fail example startup before partial player creation
- tests that guard the example's scene config consumption boundary
- docs that explain which example content is data-driven and which runtime content stays code-driven

---

## Scope

This version includes:

- `createDodgeBlocksSceneConfig(...)` for viewport-aware static scene data
- example asset preloading through the config's `assets` section
- player transform / size / collider creation through scene config bootstrap
- README/example docs updates
- Node smoke tests that check the example consumes scene config APIs

This version does not include:

- moving dynamic hazard spawning into scene config
- serializing runtime gameplay state
- editor or inspector mutation controls
- visual authoring tools
- save/load systems
- new gameplay mechanics

---

## Semantics

- Static boot-time content can be represented as scene config.
- Runtime-random gameplay content remains code-driven.
- The example should use package-style imports just like a downstream game.
- The example should validate scene config before creating static entities.
- Asset preloading remains async and browser-facing through the existing manifest loader.

---

## Done When

- `dodge-blocks` uses scene config for static asset/entity declarations.
- Player base components are bootstrapped through `bootstrapSceneFromConfig(...)`.
- Example tests guard against regressing back to purely hand-built player base data.
- Documentation makes the engine-package/not-editor boundary explicit.
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass.
