# leaferGame v0.14.0

## Version Positioning

`v0.14.0` starts the next stage:

**Runtime Observability Hardening Sprint**

The previous stage made input semantics reusable through Node-safe action maps, keyboard binding normalization, example integration, read-only tooling visibility, and package-facing boundary docs.

This stage strengthens the engine package's runtime observability layer without turning tooling into an editor.

---

## Why This Stage

The project already has:

- stable ECS-style scene/entity/component/system primitives
- package-facing `/core`, `/framework`, and `/tooling` entrypoints
- read-only debug snapshots
- scene/entity inspector snapshots
- asset, game flow, sprite animation, runtime services, and input action tooling sections
- a browser tooling panel used by `examples/dodge-blocks`

The next bottleneck is not content editing. It is confidence.

As the engine grows, downstream games need a clearer way to observe what the runtime is doing: which systems are installed, which order they run in, whether they are started/enabled, and how optional runtime services and feature-specific snapshots compose.

For a reusable frontend 2D game engine package, this observability must remain deterministic, testable, and read-only.

---

## Stage Goals

`0.14.x` should move the project toward:

- clearer system/runtime status snapshots
- stable formatting for system order and lifecycle state
- package-safe tooling helpers that can be tested in Node
- browser panel readability improvements that only display state
- docs that consistently frame tooling as runtime observability, not editor UI
- regression coverage that protects read-only behavior and package boundaries

---

## Scope Boundaries

This stage should include:

- read-only runtime and system state visibility
- deterministic snapshot shapes and formatting
- package-safe exports from `/tooling` if new helpers are needed
- browser panel polish that improves readability without changing runtime state
- documentation for downstream developer consumption

This stage should not include:

- visual editor UI
- system enable/disable controls in tooling
- component value editing
- scene graph editing
- asset browser or asset management UI
- drag-and-drop scene composition
- runtime mutation buttons
- npm publishing unless explicitly decided later

---

## Done When

By the end of `0.14.x`:

- downstream games can inspect system/runtime state more clearly through read-only tooling
- tooling output remains deterministic and testable in Node
- browser panel changes, if any, are presentation-only
- docs explicitly preserve the engine-package boundary
- examples remain consumers of the engine package boundary
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass
