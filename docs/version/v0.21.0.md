# leaferGame v0.21.0

## Version Positioning

`v0.21.0` starts the next stage:

**Audio Runtime Primitives Sprint**

The previous stage closed collision query runtime primitives. This stage begins audio as a reusable engine package capability, starting from deterministic data contracts and runtime state management before any browser playback adapter is introduced.

---

## Why This Stage

The project already has reusable primitives for:

- assets and manifests
- runtime services
- input actions
- sprite animation
- scene config
- level/map metadata
- collision queries
- read-only runtime observability

Audio is another core game-engine concern. Downstream games need a reusable way to describe sound assets, semantic audio cues, channel state, volume/mute policy, and future playback adapters without hard-coding everything into examples.

This should begin as package-facing runtime contracts, not as a visual audio editor, DAW-like timeline, asset browser, waveform UI, mixing console product, or content authoring workflow.

---

## Stage Goals

`0.21.x` should move the project toward:

- Node-safe audio asset/cue/channel data contracts
- deterministic audio runtime state that tests can inspect
- simple play/stop/pause/resume intent records
- optional integration with runtime services or scene systems
- read-only tooling visibility for audio runtime state
- example consumption if the base contracts are stable enough
- package-facing docs that keep audio separate from editor or production tooling

## Current Progress

The `0.21.x` stage is starting. `v0.21.1` provides the first Node-safe audio data contract and runtime state baseline. `v0.21.2` connects that state to scene/runtime ownership without adding browser playback. `v0.21.3` should make that state visible through read-only tooling.

---

## Scope Boundaries

This stage should include:

- plain TypeScript audio data contracts
- deterministic runtime state helpers
- Node-safe tests
- package-facing exports
- read-only observability if useful
- example consumption only after the base contracts are stable

This stage should not include:

- Web Audio implementation details in the first patch
- audio file decoding
- streaming audio
- spatial audio
- waveform editing
- visual audio timelines
- mixer UI
- content authoring workflows
- asset management UI

---

## Done When

By the end of `0.21.x`:

- downstream games can describe and trigger audio intent through reusable package APIs
- audio runtime state is deterministic and testable in Node
- public docs preserve the runtime-package/not-audio-editor boundary
- optional browser/example work remains a consumer of the package API
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass
