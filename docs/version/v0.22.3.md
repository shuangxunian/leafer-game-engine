# leaferGame v0.22.3

## Version Positioning

`v0.22.3` is the third implementation pass in the `0.22.x` stage:

**Browser Audio Playback Adapter Baseline**

`v0.22.1` introduced the Node-safe playback adapter contract, and `v0.22.2` added opt-in scene/system draining integration. This version should add the first browser-facing playback adapter implementation that can consume those contracts without moving browser media details into `/framework`.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.22.3` does not add an audio editor, mixer UI, waveform editing, timeline editing, asset browser UI, audio authoring workflow, project persistence, content production workflow, spatial audio, streaming pipeline, decoding pipeline, or autoplay/permission product flow.

It only adds an opt-in browser runtime adapter that downstream games can inject into `AudioPlaybackSystem`.

---

## Goals

After `v0.22.3`, the project should support:

- a `BrowserAudioPlaybackAdapter` exported from the browser-facing `/runtime` entrypoint
- mapping audio runtime `play` operations to browser media element playback
- stop/pause/resume/volume/mute operation handling
- deterministic adapter behavior when tested with fake media elements
- clear failure messages for missing assets or missing sources
- no runtime dependency from `/framework` to DOM or browser media APIs

---

## Scope

This version should include:

- runtime-level browser audio adapter implementation
- an injectable media element factory for tests
- copied manifest lookup from `AudioRuntimeState`
- operation handling for play/stop/pause/resume/set-volume/set-muted
- tests using fake media elements instead of real sound playback
- public docs clarifying this is browser runtime integration, not audio authoring

This version should not include:

- changes to `/framework` Node-safe playback contracts
- actual bundled audio assets
- dodge-blocks playback consumption
- tooling panel changes
- editor/mixer/authoring UI
- asset browser UI
- waveform or timeline UI
- autoplay permission UX

---

## Semantics

- The adapter should be opt-in and injected by downstream runtime code.
- It should use audio manifest asset sources, not invent asset loading or authoring flows.
- It should create media elements lazily on play.
- It should keep enough instance state to stop, pause, resume, and apply channel changes.
- It should fail clearly when a play operation references a missing asset or source.
- Tests should not require a DOM, real audio files, or browser permissions.

---

## Implementation Notes

`v0.22.3` adds `BrowserAudioPlaybackAdapter` to the browser-facing runtime layer, plus:

- `BrowserAudioMediaElement` and `BrowserAudioMediaElementFactory` test seams
- lazy media element creation from `AudioRuntimeState` asset sources
- active instance tracking for stop/pause/resume operations
- channel volume and mute application for active instances
- deterministic missing asset/source errors
- direct fake-media tests that avoid real DOM playback and browser permissions

The implementation remains opt-in browser runtime integration. It does not move DOM or browser media dependencies into `/framework`, and it does not add editor, mixer, waveform, timeline, asset browser, or authoring workflows.

---

## Done When

- Browser runtime exports a playback adapter implementation.
- Fake media tests prove play/stop/pause/resume/volume/mute behavior.
- Missing asset/source failures are deterministic.
- `/framework` remains Node-safe.
- Public docs preserve the runtime-package/not-audio-editor boundary.
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass.
