# leaferGame v0.17.0

## Version Positioning

`v0.17.0` starts the next stage:

**Runtime/Game Loop Hardening Sprint**

The previous stage closed render/view contracts. This stage returns to runtime fundamentals: how `Game`, `Scene`, `World`, systems, components, and lifecycle helpers behave when downstream games hit errors, mutate entities during phases, or need clearer ownership boundaries.

---

## Why This Stage

The project already has:

- stable scene/system/component lifecycle basics
- deferred world entity mutation during update phases
- deterministic system priority and registration order
- `GameFlow`, runtime services, scheduler, and scene lifecycle helpers
- read-only runtime observability for systems and services
- render/view ownership contracts

The next bottleneck is resilience.

Downstream games need the engine to preserve runtime invariants even when a gameplay system or component throws. Errors should still surface to callers, but core phase cleanup, pending world mutations, and lifecycle flags should not silently get stuck.

This remains engine package work. It is not an editor, runtime control panel, visual debugger, devtools protocol, or app-shell framework.

---

## Stage Goals

`0.17.x` should move the project toward:

- safer runtime phase cleanup when update hooks throw
- clearer game loop error boundaries
- stronger tests around scene/world cleanup invariants
- better docs for runtime ownership and error propagation
- package-facing guidance for downstream games

## Current Progress

The `0.17.x` stage is complete:

- `v0.17.1` hardens scene/world phase cleanup when systems or components throw during runtime phases.
- `v0.17.2` clarifies `Game.tick(...)` error propagation and fixed-step accumulator behavior.
- `v0.17.3` documents runtime ownership boundaries and tests browser runtime stop/start semantics.
- `v0.17.4` closes the stage and aligns package-facing docs with the completed guarantees.

---

## Scope Boundaries

This stage should include:

- core runtime invariants
- scene/world/game loop lifecycle behavior
- tests that prove cleanup happens when hooks throw
- docs that explain which errors propagate and which cleanup is guaranteed

This stage should not include:

- visual editor UI
- runtime mutation controls
- browser devtools protocol work
- app-shell routing
- save/load systems
- network or analytics infrastructure
- npm publishing unless explicitly decided later

---

## Done When

By the end of `0.17.x`:

- scene/world phase cleanup is resilient to thrown errors
- downstream games can understand which runtime errors propagate
- docs preserve the engine-package/not-editor boundary
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass
