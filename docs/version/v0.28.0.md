# leaferGame v0.28.0

## Version Positioning

`v0.28.0` starts the next stage:

**Real Sprite / Image Rendering Sprint**

`0.27.x` closed the pointer-first puzzle interaction baseline. The next 1.0 gap is visual: examples and render nodes still rely too much on colored rectangle placeholders even though sprite assets already carry `source` metadata and browser image loading exists.

This version is a planning and boundary document. It does not add public package API by itself.

---

## Why This Stage

The engine already has:

- typed sprite asset metadata with optional `source`
- async sprite asset loading and browser image loader support
- sprite frames and animation clips
- sprite-capable render node checks
- `SpriteAnimationSystem` applying registered sprite assets to view nodes
- `attachActorSpriteView(...)` and `createTileMapLayerView(...)` creating sprite-backed render views
- examples that consume sprite assets through package-facing APIs

The missing behavior is that image-backed sprite assets should visibly render through the Leafer adapter. A downstream game should be able to declare a simple image source for a character, item, background, icon, or puzzle object and see that image reach the render layer.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.28.0` does not add a visual asset manager, atlas packer, sprite editor, image cropper, asset browser UI, CDN pipeline, bundled art library, marketplace, launcher, gallery, WeChat SDK wrapper, account system, ads, monetization, or publishing workflow.

This stage may add adapter/runtime behavior and example consumption, but it should not become a content production suite.

---

## Candidate Engine Gaps

### 1. Image-Backed Leafer Sprite Adapter

Current render sprite assets can include `source`, but the Leafer wrapper should make that source visible when possible.

Potential package-level shape:

- map `RenderSpriteAsset.source` into the Leafer render node
- preserve existing fill/width/height/cornerRadius rectangle behavior for placeholder shapes
- keep the adapter-facing contract narrow and testable

Keep out of package:

- image editing
- atlas packing
- shader/material authoring
- asset manager UI

### 2. Asset Loading To Render Asset Handoff

Asset loading and render application are currently adjacent but not fully proven end to end.

Potential package-level shape:

- make loaded/registered sprite asset data flow cleanly into render nodes
- document whether render nodes need loaded image resources or only stable source strings
- keep `/framework` Node-safe and DOM-free

Keep out of package:

- bundler plugins
- CDN policy
- remote asset validation
- production asset pipeline automation

### 3. Example Image Asset Consumption

At least one example should render real image-like assets instead of only colored rectangles.

Potential example shape:

- consume data URI or local static image-like sources through existing manifests
- prove browser build still works
- keep art choices minimal and example-owned

Keep out of package:

- bundled commercial asset pack
- image gallery
- template marketplace
- visual asset picking UI

## Done When

By the end of `0.28.x`:

- sprite assets with `source` can visibly render through the Leafer adapter
- existing fill-based placeholder rendering remains usable
- examples consume at least some image-like sprite assets through package-facing APIs
- `/framework` remains Node-safe and DOM-free
- docs explain the boundary between runtime sprite rendering and asset authoring
- no visual editor, asset manager, launcher, gallery, marketplace, SDK integration, monetization, or publishing workflow is added
- `npm run check`, `npm test`, `npm run build:example`, `npm run verify:package`, and `npm audit --audit-level=moderate` pass

The stage is closed through `v0.28.5`. The next 1.0 capability area should be
responsive Web runtime behavior, not more asset authoring surface.
