# leaferGame v0.15.0

## Version Positioning

`v0.15.0` starts the next stage:

**Data-Driven Scene Contract Hardening Sprint**

The previous stage made runtime observability clearer and better documented while keeping tooling read-only.

This stage returns to engine package fundamentals by strengthening scene config, entity template, and data-driven bootstrap contracts.

---

## Why This Stage

The project already has:

- asset manifests
- entity templates
- component schema metadata
- a minimal `SceneConfig`
- `bootstrapSceneFromConfig(...)`
- package-facing `/framework` exports
- dodge-blocks using part of the content setup through data

The next data-driven bottleneck is safety.

Downstream games need to know whether a config is structurally valid before it mutates a scene, registers assets, creates entities, or adds systems. This is important for tests, content loading, package consumers, and read-only runtime diagnostics.

For a reusable frontend 2D game engine package, this should remain plain TypeScript data-contract work. It should not become a visual editor, authoring product, asset manager, or content production workflow inside this repository.

---

## Stage Goals

`0.15.x` should move the project toward:

- Node-safe scene config validation
- clearer config diagnostics before bootstrap mutation
- stronger entity template/component validation boundaries
- safer system registry lookup semantics
- better docs for data-driven scene consumption
- example coverage where useful

## Current Progress

The `0.15.x` stage is currently closed at `v0.15.4`.

- `v0.15.1` adds a Node-safe scene config validation baseline with deterministic diagnostics and no runtime mutation.
- `v0.15.2` connects validation to bootstrap as an opt-in safety gate and adds component data diagnostics.
- `v0.15.3` makes dodge-blocks consume scene config for static asset/entity declarations.
- `v0.15.4` documents the data-driven scene contract and closes the stage.

---

## Scope Boundaries

This stage should include:

- JSON-like data shapes
- deterministic validation results
- clear validation error codes/messages
- tests for validation and mutation boundaries
- package-safe exports from `/framework`
- docs for downstream game consumption

This stage should not include:

- visual editor UI
- drag-and-drop scene editing
- component value editing UI
- asset browser or asset management UI
- runtime mutation controls
- save-file migration framework
- remote content pipelines
- npm publishing unless explicitly decided later

---

## Done When

By the end of `0.15.x`:

- downstream games can validate scene configs before mutating scenes
- invalid config diagnostics are deterministic and testable in Node
- bootstrap behavior is safer and clearly documented
- examples remain consumers of the engine package boundary
- docs preserve the engine-package/not-editor boundary
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass

---
