# leaferGame v0.12.2

## Version Positioning

`v0.12.2` is the second `0.12.x` delivery pass:

**Deterministic Runtime Scheduler**

This version adds a small, Node-safe scheduler to the `framework` entrypoint so downstream games can run delayed and repeated runtime tasks through deterministic `update(dt)` calls.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.12.2` does not add editor timelines, visual scripting, browser UI, DOM timers, async job queues, worker threads, cron-like scheduling, or npm publishing. It only adds a reusable runtime timing primitive that can be advanced by engine/game loops and tested without a browser.

---

## Goals

After `v0.12.2`, the project should support:

- scheduling one-shot delayed callbacks
- scheduling repeated interval callbacks
- explicit cancel handles
- clearing scheduled tasks
- deterministic task ordering
- catch-up behavior when a large `dt` crosses multiple interval boundaries
- safe task mutation during scheduler updates
- Node-side tests and package export coverage

---

## Scope

This version includes:

- a new framework `RuntimeScheduler`
- one-shot `schedule(...)` API
- repeated `repeat(...)` API
- scheduler task handles with active state and cancel semantics
- `update(dt)` driven timing
- tests for ordering, cancellation, clearing, repeated catch-up, and mutation during update
- README/public API updates
- `v0.12.0` stage progress update
- regression verification

This version does not include:

- scene-owned scheduler lifecycle wiring
- tooling snapshot integration
- wall-clock timers
- browser `setTimeout` / `setInterval` wrappers
- visual timeline editor
- scripting graphs
- npm publishing

---

## Semantics

- Scheduler time advances only when `update(deltaSeconds)` is called.
- One-shot tasks run once when scheduler elapsed time reaches their due time.
- Repeated tasks run in interval order and can fire multiple times in one update when `dt` crosses multiple intervals.
- Tasks with the same due time run in registration order.
- Cancelling a task prevents future callback execution.
- Tasks added during an `update(...)` call do not run until a later update.
- Invalid timing inputs fail clearly.
