# leaferGame v0.24.0

## Version Positioning

`v0.24.0` starts the next stage:

**Playable 2D Game Kit Sprint**

The product direction is a frontend 2D game engine dependency package. After the camera contract work in `0.23.x`, the next stage should move attention back from runtime observability polish to the concrete primitives that make a browser mini-game feel buildable, testable, and reusable.

This stage answers the product question: if a developer wants to remake a simple 4399-style browser game, what engine-level pieces should they be able to import instead of rewriting inside every example?

---

## Why This Stage

The engine already has a useful base:

- ECS core, scene lifecycle, systems, and deterministic update order
- input actions, keyboard/pointer button bridges, and semantic gameplay controls
- transform, size, view sync, velocity movement, collision queries, GameFlow, resources, sprite animation, audio, level layout, and camera bounds
- a browser example that proves these pieces can run together

The current gap is not "more editor tooling". The current gap is gameplay-facing convenience:

- movement helpers that avoid common arcade-control bugs
- actor/prefab-like composition helpers for player, enemy, projectile, item, and pickup patterns
- runtime HUD/game UI primitives that stay screen-space and package-facing
- tile/level rendering helpers that consume map data without becoming a level editor
- a stronger playable example that demonstrates game-making, not only engine introspection

This remains frontend 2D game engine package work, not an editor, visual authoring tool, scene builder, asset manager, level editor, or content publishing workflow.

---

## Stage Goals

`0.24.x` should move the project toward:

- reusable gameplay math and movement primitives
- actor/prefab-like gameplay composition helpers that reduce per-game boilerplate
- runtime HUD/game UI helpers for score, status, overlays, and screen-space game UI
- level/tile rendering consumption that stays runtime-only
- example improvements that prove the package can support a small playable browser game
- public docs that explain what is currently possible and what is still missing for fuller 4399-style games

## Deferred Work

`v0.23.3` camera read-only tooling visibility is intentionally deferred. It is still valid engine-package work, but it is not the highest-leverage next step for proving that this package can help build a real browser mini-game.

The camera tooling work can return later if a gameplay-facing stage needs camera state inspection. Until then, `0.24.x` should prioritize runtime primitives and example consumption that directly improve playability.

---

## Scope Boundaries

This stage should include:

- plain TypeScript gameplay helpers
- framework primitives usable by multiple downstream games
- example consumption that removes local one-off gameplay logic
- Node-safe tests for math, state, and contracts
- browser example regression where useful
- docs that frame the project as a game engine dependency package

This stage should not include:

- visual editors
- drag-and-drop scene building
- asset browser or asset manager UI
- level authoring tools
- entity/component mutation panels
- save/load project workflows
- publishing pipelines
- editor-first feature planning

---

## Done When

By the end of `0.24.x`:

- common arcade movement behavior is provided by framework helpers
- common actor templates can be composed through framework helpers without introducing editor prefab authoring
- simple HUD text nodes can be created through framework helpers without introducing UI authoring
- tile map runtime data can be consumed as simple world-space render layers without introducing map authoring
- examples consume gameplay-facing framework primitives instead of local one-off helpers where practical
- the README explains the current "can build small browser games, not yet full commercial engine" status clearly
- package-boundary tests keep the playable-game direction visible
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass
