# leaferGame v0.22.4

## Version Positioning

`v0.22.4` closes the `0.22.x` stage:

**Audio Playback Example Consumption and Boundary Closeout**

`v0.22.1` introduced the Node-safe playback adapter contract, `v0.22.2` added opt-in scene/system draining integration, and `v0.22.3` added the first browser runtime playback adapter. This version makes `examples/dodge-blocks` consume that playback path as a downstream-style browser example, then documents the completed audio playback boundary.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.22.4` does not add an audio editor, mixer UI, waveform editing, timeline editing, asset browser UI, audio authoring workflow, project persistence, content production workflow, spatial audio, streaming pipeline, decoding pipeline, bundled sound library, or autoplay/permission product flow.

The example may opt into browser playback through package APIs, but it must not become an audio editor, mixer, asset manager, or content authoring surface.

---

## Goals

After `v0.22.4`, the project supports:

- `examples/dodge-blocks` installing `AudioPlaybackSystem` with `BrowserAudioPlaybackAdapter`
- example audio manifest assets carrying explicit runtime `source` values
- gameplay audio intent draining into the browser adapter without moving browser dependencies into `/framework`
- example docs explaining playback as opt-in runtime consumption, not editor/mixer scope
- README/public API/version progress updates marking `0.22.x` as closed
- package-boundary tests proving the example consumes playback APIs while preserving non-editor/audio-authoring boundaries

---

## Scope

This version includes:

- example-level browser playback adapter consumption
- example-level placeholder audio sources suitable for runtime manifest consumption
- package-style imports for framework/runtime APIs
- example README updates
- README/public API/roadmap/stage closeout updates
- package-boundary tests for example consumption and docs

This version does not include:

- new framework playback contracts
- new browser audio product UX
- Web Audio graph management
- audio decoding or streaming pipelines
- real bundled sound asset production
- playback buttons, volume sliders, mixer controls, waveform/timeline UI, or asset browser UI
- editor or authoring workflows

---

## Semantics

- Playback remains opt-in and example-owned.
- The browser adapter is injected from the browser-facing runtime layer.
- The Node-safe `/framework` entrypoint remains free of DOM and browser media dependencies.
- Audio manifest `source` values are runtime asset references, not an asset authoring pipeline.
- Tooling remains read-only observability. It can show audio runtime intent records, but does not mutate playback, channels, assets, entities, or scene config.
- Autoplay and permission behavior are browser concerns for downstream apps. This version should not add product UX for those flows.
