# leaferGame v0.28.2

## Version Positioning

`v0.28.2` adds the second implementation slice for the real sprite / image rendering stage:

**Asset Loading To Render Asset Handoff**

`v0.28.1` made the Leafer adapter able to render image-backed sprite assets. This version makes the framework handoff path explicit so registered sprite asset metadata can be passed to render nodes without leaking registry-only state.

---

## What Changed

- `AssetRegistry.getSpriteRenderAsset(id)` returns a copied `RenderSpriteAsset` for optional render handoff.
- `AssetRegistry.requireSpriteRenderAsset(id)` returns the same copied render contract and fails clearly for missing sprites.
- `SpriteAnimationSystem` now applies render asset copies to sprite-capable view nodes.
- `attachActorSpriteView(...)` can resolve `assets + assetId` into a render asset before attaching the view.
- Source-backed animation frames and actor sprite views are covered by framework tests.

The render handoff passes stable sprite metadata and `source` strings. Browser image loading can still be used to preload/check assets, but render nodes do not require a DOM image object from the framework layer.

---

## Product Boundary

This is framework/runtime handoff work.

`v0.28.2` does not add a visual asset manager, atlas packer, sprite editor, image cropper, asset browser UI, bundled asset library, CDN pipeline, bundler plugin, launcher, gallery, marketplace, WeChat SDK wrapper, account system, ads, monetization, or publishing workflow.

The next `0.28.x` slice should prove example image asset consumption through package-facing APIs.
