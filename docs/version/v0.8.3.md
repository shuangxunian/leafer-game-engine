# leaferGame v0.8.3

## Version Positioning

`v0.8.3` is the third `0.8.x` delivery pass:

**Browser Image Sprite Loader Adapter**

This version connects the asset loading abstraction to the browser's native image loading path while keeping the engine package boundary clear.

---

## Product Boundary

This repository remains a frontend 2D game engine dependency package.

`v0.8.3` adds a runtime/browser adapter helper that game projects can call when loading sprite images. It does not add an editor, asset panel, drag-and-drop resource workflow, or any visual authoring surface.

Future editor-like products should live above this package and consume its runtime, schema, snapshot, and asset APIs.

---

## Goals

After `v0.8.3`, the project should support:

- loading sprite assets through browser `Image`
- clear errors for sprites without image sources
- deterministic image load and error behavior
- optional image configuration such as `crossOrigin`
- testability without requiring a real DOM image implementation

---

## Scope

This version includes:

- a browser image sprite loader factory
- a small injectable image abstraction for tests and non-standard browser hosts
- source validation before assigning image `src`
- success and failure tests
- integration with existing `AssetRegistry.loadSprite(...)`

This version does not include:

- editor asset browser UI
- drag-and-drop importing
- texture atlas packing
- retry policies
- loading progress percentages
- image cache eviction
- audio loading

---

## Semantics

- The loader requires a sprite `source`.
- The loader resolves when the browser image reports `load`.
- The loader rejects when the browser image reports `error`.
- The loader can set `crossOrigin` before assigning `src`.
- Tests can inject a fake image constructor so behavior stays deterministic in Node.
- The loader remains a framework utility and does not mutate rendering nodes directly.

---

## Done When

- A browser image sprite loader helper exists and is exported from `framework`.
- Missing-source, load-success, and load-error paths are tested.
- Existing async manifest loading behavior remains compatible.
- `npm run check`, `npm test`, and `npm run build:example` pass.
