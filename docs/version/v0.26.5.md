# leaferGame v0.26.5

## Version Positioning

`v0.26.5` closes the `0.26.x` framework extraction stage:

**Framework Extraction Closeout From Two Playable Examples**

`0.26.x` started with a simple rule: do not invent large abstractions from taste. Use `examples/dodge-blocks` and `examples/collect-stars` as evidence, then extract only the small runtime/framework helpers that clearly belong in the engine package.

That stage is now complete at the baseline level.

No new public package API is added in this closeout.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.26.5` does not add a visual editor, scene builder, prefab authoring UI, spawn editor, asset browser, example launcher, gallery, marketplace, save/load project workflow, account system, leaderboard, analytics service, monetization, or publishing workflow.

It closes documentation and package-boundary coverage for the framework extraction stage.

---

## Stage Outcome

After `0.26.x`, the engine package has four small helpers extracted from repeated example friction:

- `clampPositionToBounds(...)`
  - Pure runtime movement math for keeping a top-left position inside rectangular bounds while accounting for size and padding.
- `attachActorSpriteView(...)`
  - Runtime view attachment helper for connecting an existing ECS entity to a sprite-backed `ViewComponent`.
- `randomPositionInBounds(...)`
  - Pure runtime placement math for selecting a top-left position inside rectangular bounds while accounting for size and padding.
- Example-owned `getGameplaySnapshot()` convention
  - Both playable examples expose small read-only gameplay snapshots, but the gameplay snapshot pattern remains example-owned and the engine package does not own a generic gameplay snapshot API.

Together these prove the project can now use playable examples to pressure-test public package APIs, identify repeated friction, and extract small reusable runtime helpers without turning the repository into an editor or one-size-fits-all gameplay framework.

---

## What Stayed Example-Owned

The following remains intentionally outside `/framework`:

- concrete player controllers
- game-specific input action ids
- scoring and timer rules
- enemy behavior
- collectable behavior
- spawn cadence
- random size and speed decisions
- pause/resume content copy
- best score persistence
- level semantics beyond runtime data helpers
- gameplay debugger UI or mutable inspector state

These may be revisited only after more examples prove a repeated package-generic need. Until then, they belong to downstream games.

---

## What We Did Not Build

`0.26.x` did not add:

- a generic player controller component
- a generic spawn system
- a prefab format
- a scoring framework
- a timer framework
- a random service or seeded RNG runtime
- collision avoidance placement
- map-aware placement
- a gameplay debugger
- an editor, launcher, gallery, marketplace, or content workflow

That restraint is the point of the stage: the package gained reusable primitives without swallowing game-specific rules.
