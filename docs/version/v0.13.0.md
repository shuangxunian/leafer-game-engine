# leaferGame v0.13.0

## Version Positioning

`v0.13.0` starts the next stage:

**Input Actions And Control Mapping Sprint**

The previous stage made runtime coordination services reusable through deterministic events, scheduling, scene integration, and read-only tooling visibility.

This stage returns to a core gameplay concern: input should be expressed as reusable game actions instead of hard-coded physical keys in every downstream project.

---

## Why This Stage

The project already has:

- a framework `InputSystem`
- a browser keyboard bridge
- reusable gameplay flow and runtime services
- examples that consume engine APIs through package-style imports

The next input bottleneck is semantic mapping.

Games usually want logic such as `move:left`, `jump`, `confirm`, or `pause`, while physical inputs can vary across keyboard layouts, platforms, accessibility preferences, and future gamepad/touch adapters.

For a reusable frontend 2D game engine package, input actions should be Node-safe, deterministic, and independent from any editor or settings UI.

---

## Stage Goals

`0.13.x` should move the project toward:

- a Node-safe input action mapping model
- keyboard binding normalization and validation
- action lookup helpers for pressed and just-pressed states
- browser bridge compatibility with existing `InputSystem`
- optional example integration where useful
- read-only tooling visibility for input state only if it stays runtime-observability focused
- docs that frame input mapping as an engine package API, not an editor keybinding UI

---

## Scope Boundaries

This stage should include:

- pure TypeScript input mapping primitives
- deterministic behavior that can be tested in Node
- package-safe exports from `/framework`
- browser keyboard compatibility through existing runtime paths
- documentation for downstream game consumption

This stage should not include:

- visual editor UI
- keybinding settings screens
- user profile persistence
- gamepad support unless explicitly scoped later
- touch gesture systems unless explicitly scoped later
- DOM-dependent APIs as the source of truth
- npm publishing unless explicitly decided later

---

## Done When

By the end of `0.13.x`:

- downstream games can define semantic actions separately from physical keyboard keys
- action pressed / just-pressed behavior is deterministic and testable
- browser keyboard input remains compatible with existing examples
- public docs explain the input action boundary
- examples remain consumers of the engine package boundary
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass
