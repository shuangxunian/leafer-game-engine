# leaferGame v0.9.0

## Version Positioning

`v0.9.0` starts the next stage:

**Game Flow And Scene Lifecycle Sprint**

The previous stage completed the first resource loading baseline. This stage focuses on reusable game-flow primitives: start, run, pause, resume, end, reset, and the surrounding lifecycle rules that most small games need.

---

## Why This Stage

Current examples already contain gameplay flow logic, but that logic still lives inside the sample game system:

- start screen
- running gameplay
- pause/resume
- gameover
- restart/reset
- HUD updates driven by phase changes

For a reusable frontend 2D game engine package, these concepts should not be reinvented by every game. The engine should provide small, testable primitives that examples and downstream projects can consume.

---

## Stage Goals

`0.9.x` should move the project toward:

- reusable game-flow state primitives
- explicit transition semantics for common gameplay phases
- lifecycle-friendly pause/resume/end/reset behavior
- debug/tooling visibility for game-flow state
- eventual dodge-blocks migration from local flow code to framework flow primitives

## Scope Boundaries

This stage should include:

- pure framework-level flow primitives
- deterministic transition results
- tests for allowed and disallowed transitions
- example integration as a consumer
- tooling/debug visibility where useful

This stage should not include:

- menu UI widgets
- save systems
- level editor flows
- visual scene editor UI
- routing or app-shell framework integration
- multiplayer session state

---

## Done When

By the end of `0.9.x`:

- common gameplay phases have a reusable framework primitive
- examples can use the primitive instead of hand-rolled local phase logic
- flow state can be inspected where useful
- lifecycle semantics are documented
- `npm run check`, `npm test`, and `npm run build:example` pass

---

## Relation to Previous Stages

- `0.5.x` introduced data-driven content primitives.
- `0.6.x` and `0.7.x` made runtime state inspectable and interactive.
- `0.8.x` made asset loading more production-ready.
- `0.9.x` should make gameplay flow reusable instead of sample-specific.
