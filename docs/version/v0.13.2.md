# leaferGame v0.13.2

## Version Positioning

`v0.13.2` is the second `0.13.x` delivery pass:

**Input Action Example Integration**

This version connects the `v0.13.1` input action mapping API to the browser example so gameplay code consumes semantic actions instead of hard-coded physical keyboard keys.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.13.2` does not add editor UI, keybinding settings screens, persistence, gamepad adapters, touch gestures, browser control panels, or npm publishing. It only proves that downstream-style example gameplay can use the engine package's input action APIs.

---

## Goals

After `v0.13.2`, the project should support:

- a dodge-blocks input action map for movement, confirm/start, and pause/resume
- player movement reading semantic actions such as `move:left` instead of physical keys
- game-flow controls reading semantic actions such as `confirm` and `pause`
- example docs that describe controls as action mappings
- tests that guard the example against direct physical-key gameplay queries

---

## Scope

This version includes:

- example-level input action definitions
- `PlayerControllerComponent` consuming `InputActionMap`
- `DodgeGameSystem` consuming `InputActionMap`
- dodge-blocks docs updates
- package-boundary/example tests for action-map consumption
- `v0.13.0` stage progress update
- regression verification

This version does not include:

- new framework APIs
- browser keybinding UI
- keybinding persistence
- gamepad or touch support
- input tooling panel integration
- gameplay feature changes
- npm publishing

---

## Semantics

- The example remains a downstream consumer of package-style imports.
- Physical keyboard keys stay declared in one action-map file.
- Gameplay systems read action ids, not raw keyboard keys.
- Browser keyboard input still flows through `BrowserKeyboardBridge` and `InputSystem`.
- Action mapping remains read-only during gameplay.

---
