# leaferGame v0.18.0

## Version Positioning

`v0.18.0` starts the next stage:

**Level/Map Runtime Primitives Sprint**

The previous stage closed runtime/game loop hardening. This stage moves into reusable level and map primitives: tile map data contracts, coordinate helpers, spawn/region concepts, and later scene-level level loading boundaries.

---

## Why This Stage

The project already has:

- stable `Game`, `Scene`, `World`, component, system, and runtime phase behavior
- scene config validation and safe bootstrap
- render/view contracts and camera mapping
- asset manifests and sprite animation contracts
- input actions and game-flow primitives
- read-only runtime observability

The next gameplay-facing bottleneck is level structure.

Downstream games need a reusable way to describe map-shaped content and convert between tile/grid coordinates and world coordinates without hard-coding those helpers inside every example. This should start as plain TypeScript runtime data contracts, not as a visual map editor or content authoring workflow.

---

## Stage Goals

`0.18.x` should move the project toward:

- Node-safe tile map data contracts
- deterministic tile/world coordinate helpers
- spawn point and region data primitives
- clearer level/map docs for downstream games
- optional scene config integration when the primitives are stable
- tests that prove validation, copying, and coordinate behavior

---

## Scope Boundaries

This stage should include:

- plain TypeScript data contracts
- deterministic validation behavior
- Node-safe coordinate helpers
- docs that describe runtime usage
- tests for map/level primitives

This stage should not include:

- visual tile map editor UI
- drag-and-drop level editing
- asset browser or map asset management UI
- painting tools
- pathfinding systems
- physics engines
- save/load or publishing workflows
- npm publishing unless explicitly decided later

---

## Done When

By the end of `0.18.x`:

- downstream games can use engine-provided tile/map primitives instead of per-game helpers
- tile/world coordinate behavior is documented and tested
- level/map data remains package-facing runtime data, not editor state
- examples remain consumers of the engine package boundary
- docs preserve the engine-package/not-editor boundary
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass
