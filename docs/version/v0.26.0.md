# leaferGame v0.26.0

## Version Positioning

`v0.26.0` starts the next stage:

**Framework Extraction From Two Playable Examples Sprint**

`0.24.x` proved the first playable loop through `examples/dodge-blocks`.
`0.25.x` proved the engine package can support a second gameplay shape through `examples/collect-stars`.

The next stage should use those two examples as evidence. The goal is not to invent large abstractions from taste. The goal is to identify repeated runtime/gameplay friction that appears in both examples, then extract only the small helpers that clearly belong in the engine package.

---

## Why This Stage

The project now has two downstream-style browser examples with different loops:

- `dodge-blocks`
  - survival loop, hazards, pause/resume, best score, audio, tooling panel, level config
- `collect-stars`
  - collection loop, countdown timer, restart, actor templates, bounded movement, tile-backed playfield

They overlap in useful ways:

- semantic movement actions
- normalized directional movement
- bounded player movement
- phase-driven gameplay loops
- score/timer HUD updates
- actor/template based ECS setup
- runtime spawning
- collision-driven gameplay outcomes
- optional read-only gameplay snapshots

That overlap is strong enough to start a framework extraction stage. It is not strong enough to justify a broad game template, visual authoring system, prefab editor, example marketplace, or one-size-fits-all gameplay framework.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.26.0` is a planning and boundary document. It does not add public framework API by itself.

This stage may add small runtime/framework helpers later, but only when they satisfy all of these rules:

- the need appears in both playable examples or is clearly package-generic
- the helper is runtime code or data contract, not content authoring
- examples remain downstream consumers
- the helper does not encode one game's scoring, enemies, levels, assets, UI copy, or win/loss rules
- docs and package-boundary tests explain why it belongs in the package

This stage should not add:

- visual editor UI
- scene graph editing
- prefab/actor authoring UI
- example launcher or game gallery product
- template marketplace
- asset manager
- save/load project workflow
- leaderboard, accounts, monetization, or publishing workflow
- large gameplay framework abstractions before small helpers prove themselves

## Candidate Extraction Areas

### 1. Bounded Directional Movement

Both examples move a player with semantic directional input, normalized movement, speed, size, and bounds.

Potential package-level shape:

- data helpers for movement bounds
- clamping helpers that account for entity size and optional padding
- possibly a small component only if it can avoid game-specific action ids

Keep out of package:

- game-specific controls
- player art
- dodge/collect scoring rules
- hardcoded viewport assumptions

### 2. Gameplay Loop State Helpers

Both examples use `GameFlow` around ready/running/ended-style phases. `dodge-blocks` also has pause/resume and best score.

Potential package-level shape:

- docs and patterns first
- small helper only if repeated phase transitions become mechanical
- stronger snapshot convention for example-owned game systems

Keep out of package:

- one fixed game loop class
- hardcoded score/timer rules
- win/lose content copy

### 3. Actor And Runtime Spawn Helpers

Both examples create actors from transform/size/collider/view concepts. `collect-stars` now uses actor templates for player/star data; `dodge-blocks` uses templates/factories for player/hazards.

Potential package-level shape:

- narrower actor factory convenience
- spawn helpers for template + render node wiring
- stronger docs on when to use templates vs factories vs scene config

Keep out of package:

- visual prefab editor
- hierarchy editor
- content browser
- project file format

### 4. HUD And Gameplay Snapshot Conventions

Both examples update score/status HUD text. `dodge-blocks` exposes a read-only gameplay snapshot; `collect-stars` may eventually benefit from the same pattern.

Potential package-level shape:

- documented example-owned snapshot convention
- tiny helper types only if repeated
- optional tooling consumption guidance without mutating gameplay

Keep out of package:

- generic gameplay debugger UI
- score persistence
- leaderboards
- account services

---

## Stage Closeout

The `0.26.x` stage extracted small helpers only:

- `clampPositionToBounds(...)`
- `attachActorSpriteView(...)`
- `randomPositionInBounds(...)`
- an example-owned read-only gameplay snapshot convention

The stage intentionally did not extract game-specific player controllers, scoring/timer systems, spawn cadence, enemy behavior, collectable behavior, prefab authoring, editor UI, launcher/gallery surfaces, persistence, or content workflows.

---

## Done When

By the end of `0.26.x`:

- duplicated gameplay helper candidates are audited against both examples
- at least one small extraction candidate is accepted or explicitly rejected with reasons
- examples still import through package-style entrypoints
- no editor, launcher, gallery, marketplace, or authoring workflow is added
- README and public API docs explain the extraction boundary
- package-boundary tests cover the stage docs and non-editor positioning
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass

The `0.26.x` stage is complete.
