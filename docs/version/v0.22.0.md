# leaferGame v0.22.0

## Version Positioning

`v0.22.0` starts the next stage:

**Audio Playback Adapter Sprint**

`0.21.x` closed audio runtime primitives: downstream games can define audio manifests, install scene-owned audio runtime state, record deterministic intent operations, inspect those operations through read-only tooling, and validate example consumption without playback.

This stage should connect audio intent to pluggable playback adapters, starting from contracts and deterministic operation draining before adding any browser-specific implementation.

---

## Why This Stage

The engine now has reusable audio intent state, but downstream browser games eventually need a package-level bridge from recorded intent to actual playback.

The next step should not jump directly into a hard-coded Web Audio implementation inside an example. Instead, it should first establish:

- a small playback adapter interface
- deterministic operation draining from `AudioRuntimeState`
- explicit adapter result/error records
- a scene/system integration path that remains testable without DOM
- browser playback work only after the contract is stable

This remains game engine package work, not an editor, mixer, DAW, waveform UI, or content authoring workflow.

---

## Stage Goals

`0.22.x` should move the project toward:

- Node-safe audio playback adapter contracts
- deterministic draining of audio runtime operations
- playback result/error reporting that tooling can inspect later
- optional scene system integration for advancing playback adapters
- browser adapter implementation only after the Node-safe contract is tested
- example consumption only after playback behavior is explicit and optional
- package-facing docs that keep playback separate from audio authoring tools

---

## Scope Boundaries

This stage should include:

- plain TypeScript playback adapter contracts
- deterministic operation drain helpers
- Node-safe tests
- package-facing exports
- explicit playback result/error data
- browser adapter work only after contract tests exist
- example consumption only as opt-in package API usage

This stage should not include:

- visual audio editors
- waveform editing
- timeline editing
- mixer UI
- volume sliders as authoring controls
- asset browser UI
- audio project files
- content production workflows
- hidden playback side effects during tests

---

## Done When

By the end of `0.22.x`:

- downstream games can connect audio intent to a playback adapter through reusable package APIs
- adapter behavior is deterministic and testable without DOM
- browser playback remains opt-in
- public docs preserve the runtime-package/not-audio-editor boundary
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass
