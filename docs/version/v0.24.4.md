# leaferGame v0.24.4

## Version Positioning

`v0.24.4` is the fourth implementation pass in the `0.24.x` playable game kit stage:

**Tile Map Layer View Baseline**

This version adds a small runtime helper for consuming existing `TileMap` data as visible world-space render nodes. The goal is to let downstream games turn map-shaped runtime data into a simple visual layer without writing the same tile loop in every project, and without turning this repository into a tile map editor or content authoring tool.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.24.4` does not add a tile map editor, painting tools, drag-and-drop level authoring, map asset manager, tileset browser, collision generation, pathfinding, save/load workflow, or content publishing pipeline.

It only adds code-first runtime consumption of already-defined tile map data.

---

## Goals

After `v0.24.4`, the project should support:

- a Node-safe tile map layer view helper
- creating one render container for a tile map layer
- creating sprite nodes for non-empty tile ids
- positioning tile sprites through existing tile/world coordinate helpers
- optional tile asset resolution for fill/source/size metadata
- attaching the layer to the world render layer by default
- dodge-blocks playfield tile map visual consumption
- tests proving skipped empty tiles, sprite positioning, asset application, and target layer attachment
- docs preserving the engine-package/not-editor boundary

---

## Scope

This version includes:

- `createTileMapLayerView(...)` in `framework`
- public export and package-boundary coverage
- dodge-blocks tile layer visual consumption
- Node tests using fake render adapter and render scene objects
- README and public API wording for runtime tile map consumption

This version does not include:

- tile painting
- visual map editing
- tileset management UI
- map-to-collider generation
- chunking, culling, batching, or spatial optimization
- autotiling
- pathfinding
- editor or content production workflows

---

## Semantics

- The helper reads an existing `TileMap`; it does not mutate tile data.
- Empty `null` tiles are skipped.
- Non-empty tile ids create sprite nodes positioned at `tileMap.tileToWorld(...)`.
- Sprite width/height should match tile size unless a downstream adapter or asset overrides rendering internally.
- The returned container is attached to the `world` render layer by default.
- Callers can provide a tile asset resolver to map tile ids to richer render sprite assets.
