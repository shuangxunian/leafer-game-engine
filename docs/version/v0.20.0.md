# leaferGame v0.20.0

## Version Positioning

`v0.20.0` starts the next stage:

**Collision Query Runtime Primitives Sprint**

The previous stage closed pointer/input runtime primitives. This stage strengthens collision as a reusable runtime data surface by moving beyond boolean/entity-list checks toward stable collision pair queries that gameplay, tests, and future read-only tooling can consume.

---

## Why This Stage

The project already has:

- `ColliderComponent`
- `CollisionSystem`
- AABB intersection checks
- layer filtering
- `enter`, `stay`, and `exit` semantics
- collider visualization through tooling
- dodge-blocks gameplay consumption

The next collision bottleneck is structured query data.

Downstream games often need to know not only that an entity collided, but which pair collided, which layers were involved, and which rects produced the contact. This should start as deterministic runtime query data, not as a physics engine, collision editor, trigger authoring UI, or map-generated collider workflow.

---

## Stage Goals

`0.20.x` should move the project toward:

- stable collision pair data contracts
- current / enter / stay / exit pair queries
- defensive snapshot copies for collision rects
- Node-safe tests for pair semantics and lifecycle
- public docs that frame collision as runtime query data
- optional read-only tooling visibility if useful later

## Scope Boundaries

This stage should include:

- plain TypeScript collision data contracts
- deterministic AABB pair snapshots
- current / enter / stay / exit query helpers
- defensive copies
- Node-safe tests
- package-facing docs

This stage should not include:

- rigid bodies
- physics simulation or collision resolution
- bounce/friction/mass/forces
- swept collision or continuous collision detection
- spatial partitioning unless explicitly scoped later
- editor collider handles
- map-generated colliders
- visual collider authoring
- content production workflows

---

## Done When

By the end of `0.20.x`:

- downstream games can inspect structured collision pair data
- pair query output is deterministic and copied safely
- collision docs preserve the runtime-package/not-physics-editor boundary
- existing collision behavior remains compatible
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass
