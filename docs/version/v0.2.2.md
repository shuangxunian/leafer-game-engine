# leaferGame v0.2.2

## 版本定位

`v0.2.2` 是 `0.2.x` 阶段里的第二个落地批次。

如果说：

- `0.2.1` 更偏 core/framework 的复用基础能力

那么这一版更偏：

- sample 结构整理
- render 挂载边界清晰化
- 状态流抽象成可复用能力

可以把它理解为：

**Structure and Flow Pass**

---

## 版本目标

`v0.2.2` 完成后，至少要做到：

- sample 不再把 world、UI、overlay 都直接塞进同一个 render root
- runtime 渲染树具备最小 layer 结构
- `dodge-blocks` 的 phase 管理不再是 example 私有散落逻辑
- framework 中开始出现可复用的状态流抽象

---

## 范围边界

这一版要做：

- render layer 拆分
- sample layer 接入
- 轻量 `StateMachine`
- sample phase 迁移

这一版不做：

- camera 对 layer 的真正驱动
- 多 layer parallax
- 更复杂的状态图编辑能力
- 业务级事件总线

---

## 任务清单

### 1. RenderScene 增加基础 layers

目标：

- 给渲染树一个最小但清楚的结构骨架

建议 layers：

- `background`
- `world`
- `ui`
- `overlay`

完成标准：

- `RenderScene` 暴露稳定的 `layers`
- adapter 默认会创建这些 layer 容器
- 旧的 `root` 仍然保留为总根节点

---

### 2. Sample 改为按 layer 挂载

目标：

- 把不同职责的节点挂到合适层里

建议挂载方式：

- player / hazard -> `world`
- score / status / title -> `ui`
- start / pause / gameover overlay -> `overlay`

完成标准：

- sample 中不再把所有节点直接挂在 `renderScene.root`
- 后续 camera 和 debug overlay 有自然插口

---

### 3. 提供轻量 StateMachine

目标：

- 把“状态切换”抽成 framework 可复用能力

建议能力：

- `getState()`
- `is()`
- `matches()`
- `transition()`
- `onEnter` / `onExit` / `onTransition`

完成标准：

- 状态流抽象足够轻，不把 framework 变重
- 当前 sample 能直接消费
- 下一份 sample 可以复用这套模型

---

### 4. 迁移 dodge-blocks phase 管理

目标：

- 用通用状态机替掉 sample 内部的手写 phase 逻辑

完成标准：

- sample 仍然保持原有玩法行为
- HUD 更新和 phase 切换边界更清楚
- “开始 / 运行 / 暂停 / 结束”不再只是 example 私有实现细节

---

## 验收清单

发布 `v0.2.2` 前，至少确认：

- renderScene 是否已有明确 layers
- sample 是否已按 layer 挂载节点
- sample 是否已切到 framework 状态流
- 状态切换是否已有最小测试覆盖

---

## 与 v0.2.0 的关系

这版不是新阶段，而是 `v0.2.0` 目标下的 patch 批次。

关系可以理解为：

- `0.2.1` 完成调度、查询、依赖校验
- `0.2.2` 完成结构层和状态流抽象

做完这两步，`0.2.x` 才比较像一个完整的 “Reusability Sprint”。
