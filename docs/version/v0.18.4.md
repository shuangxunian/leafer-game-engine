# leaferGame v0.18.4

## Version Positioning

`v0.18.4` is the fourth implementation pass in the `0.18.x` stage:

**Dodge Blocks Level Config Consumption**

This version makes `examples/dodge-blocks` consume the scene config level/map declaration path added in `v0.18.3`. The example should prove that a downstream-style browser game can declare level metadata, bootstrap it through the engine package API, and use the resulting runtime helpers without turning the repository into a level editor.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.18.4` does not add a visual map editor, drag-and-drop level authoring, map rendering, collision generation, entity spawning from map tiles, asset management UI, save/load workflows, or tooling mutation controls. It only updates an example consumer to use existing runtime data contracts.

The example remains a consumer and integration test, not the engine product itself.

---

## Goals

After `v0.18.4`, `examples/dodge-blocks` should:

- declare a minimal `level.tileMap` in scene config
- declare level layout metadata for player spawn and playfield bounds
- bootstrap level declarations through `bootstrapSceneFromConfig(...)`
- read the returned `TileMap` and `LevelLayout` helpers
- use level layout metadata to configure player movement bounds and reset position
- keep dynamic hazards code-driven because they depend on runtime randomness
- document the example as package consumption, not editor functionality

---

## Scope

This version includes:

- dodge-blocks scene config level/map declarations
- example runtime consumption of bootstrapped level helpers
- focused package-boundary tests for example consumption
- README/example/public API progress updates

This version does not include:

- new engine runtime primitives
- level/map tooling panels
- map renderer
- collision or trigger generation
- editor or authoring workflows

---

## Semantics

- The player spawn remains static boot-time metadata.
- The playfield region describes movement bounds for this example.
- The tile map is deliberately minimal and proves data-contract consumption only.
- Hazard spawning remains code-driven and randomized at runtime.
- Level declarations do not mutate scene content by themselves; the example explicitly decides how to consume the returned helpers.

---

## Done When

- `examples/dodge-blocks` declares and consumes scene config level/map data.
- The example still uses package-style imports.
- Tests prove the example consumes level helpers without importing editor/authoring concepts.
- README/public API/example docs mention `v0.18.4` progress.
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass.
