# leaferGame v0.19.4

## Version Positioning

`v0.19.4` closes the `0.19.x` stage:

**Pointer/Input Runtime Boundary Closeout**

The previous patches added pointer button action bindings, browser pointer button bridging, tooling formatting, and dodge-blocks example consumption. This version packages that work into a clearer package-facing boundary so downstream games can understand how to use pointer input primitives without mistaking them for an input settings product or editor control surface.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.19.4` does not add pointer position tracking, pointer move state, drag/drop behavior, gesture recognition, hit testing, visual rebinding UI, settings screens, local storage profiles, editor controls, or content authoring workflows. It only documents and tests the reusable runtime input boundary that already exists.

Pointer input remains a runtime input contract, not an editor interaction model.

---

## What Is Now Included

The closed `0.19.x` stage includes:

- `PointerButton` values for `primary`, `secondary`, and `auxiliary`
- `definePointerButtonBinding(...)`
- `normalizePointerButton(...)`
- `getPointerButtonInputId(...)`
- `InputActionMap` support for pointer button bindings
- read-only tooling formatting for pointer bindings
- `BrowserPointerButtonBridge`
- browser button number mapping for primary, auxiliary, and secondary buttons
- cleanup behavior for `pointerup`, `pointercancel`, `blur`, and `detach()`
- dodge-blocks example consumption through semantic `confirm`

---

## Boundary Summary

Pointer input in this package means:

- low-level button state written into `InputSystem`
- semantic action bindings through `InputActionMap`
- browser adapter code that maps DOM pointer button numbers into engine input ids
- read-only observability through existing input action tooling
- downstream-style example consumption

Pointer input in this package does not mean:

- cursor or touch position tracking
- drag/drop or gesture authoring
- hit-test driven UI controls
- input rebinding screens
- persistent settings/profile storage
- editor selection, scene editing, or content authoring

Future pointer stages can add position, gesture, or richer interaction APIs only if they remain reusable runtime package APIs and preserve the not-editor boundary.
