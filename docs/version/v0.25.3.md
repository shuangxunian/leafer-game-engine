# leaferGame v0.25.3

## Version Positioning

`v0.25.3` is the third implementation pass in the `0.25.x` second playable example stage:

**Collect Stars Package API Consumption Hardening**

`v0.25.2` made `examples/collect-stars` playable. This version hardens how that example consumes the engine package by making its local game code look more like a small downstream browser game project instead of one large scene file.

The goal is not to add new framework APIs. The goal is to prove existing package APIs are enough for a second gameplay shape when the example is organized around package-facing seams.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.25.3` does not add a visual editor, example launcher product, scene builder, prefab editor, asset manager, level authoring workflow, online service, or publishing surface.

It only reorganizes and hardens example-level gameplay consumption of existing package APIs.

---

## Goals

After `v0.25.3`, `collect-stars` should:

- keep semantic input action definitions outside the scene file
- keep player movement behavior in its own component module
- keep collection/timer/gameplay-loop behavior in its own system module
- create player and star entities through `defineActorTemplate(...)` and `instantiateEntityTemplate(...)`
- keep render node creation explicit in the example, with ECS data still driven by package templates/components
- preserve package-style imports from `@shuangxunian/leafer-game-engine`
- avoid introducing collect-stars-specific framework abstractions before the example proves repeated need

---

## Scope

This version includes:

- collect-stars input action module
- collect-stars player controller module
- collect-stars actor/template helper module
- collect-stars gameplay system module
- scene file reduction to orchestration, playfield setup, HUD setup, and system wiring
- README and public API documentation updates
- package-boundary tests covering the hardened package consumption shape

This version does not include:

- hazards or enemy AI
- audio cues
- sprite animation clips
- browser tooling panel wiring
- persisted scores
- new framework helpers extracted from collect-stars
- editor, launcher, authoring, marketplace, or publishing workflow

---

## Semantics

- The playable behavior should remain equivalent to `v0.25.2`.
- Player/star ECS data should be created from actor templates.
- Scene code should remain responsible for browser/render composition.
- Example modules should import engine APIs through package subpaths, not `src`.
- Any future reusable helper should wait until both examples expose the same pain clearly.

---

## Done When

- `collect-stars` is split into package-facing gameplay modules.
- Player and star creation consume actor template APIs.
- Package-boundary tests cover the split and prevent src imports.
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass.
