# leaferGame v0.32.0

## Version Positioning

`v0.32.0` starts the next stage:

**Quick-Start Game Kit Sprint**

`0.31.x` closed the UI / dialogue / scene flow baseline. The next 1.0 gap is
fast game assembly: a downstream browser game should not need to copy the same
scene boot glue every time it wants keyboard input, pointer button input, or
pointer-local position tracking.

This version adds the first package-facing quick-start helper:

- `createSceneInputBridgeBundle(scene, options)`
- `SceneInputBridgeBundle`
- `SceneInputBridgeBundleOptions`
- `SceneInputBridgeHandle`
- `SceneInputBridgeKind`

It also extends `BrowserKeyboardBridge` with an optional injected event target,
matching the pointer bridge testability pattern while preserving the existing
browser default.

---

## Why This Stage

The engine already has enough runtime pieces for small Web games:

- `startSceneWithLifecycle(...)` for prepare / ready / running / failed startup
- `InputSystem` for low-level input state
- `BrowserKeyboardBridge`, `BrowserPointerButtonBridge`, and
  `BrowserPointerPositionBridge`
- pointer-local coordinate helpers for resized browser mounts
- HUD, dialogue, sprite, collision, audio, game flow, and scene-owned systems
- four routed examples proving arcade, collection, pointer puzzle, and dialogue
  flows

The repeated friction is now assembly glue. Multiple examples were manually
looking up `InputSystem`, creating bridges, attaching them, and wrapping
`scene.destroy()` for cleanup. That is engine-package quick-start work because
it reduces boilerplate without owning gameplay rules.

---

## Added API

`createSceneInputBridgeBundle(scene, options)` reads the scene-owned
`InputSystem`, creates the requested browser input bridges, attaches them by
default, and can detach them automatically when the scene is destroyed.

Supported bridge options:

- `keyboard: true | { target }`
- `pointerButtons: true | { target }`
- `pointerPosition: true | { target, resolver, localTarget }`
- `autoAttach`
- `detachOnSceneDestroy`

When `pointerPosition.localTarget` is provided, the helper uses
`createBrowserPointerLocalPositionResolver(localTarget)` so downstream games can
keep pointer coordinates mount-local after resize without hand-writing the
bridge setup.

`examples/collect-stars`, `examples/dialogue-choice`, `examples/pour-sort`, and
`examples/dodge-blocks` now consume this helper from the package-facing
`/framework` entrypoint.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.32.0` does not add a project generator, CLI scaffold, visual editor,
launcher, gallery, template marketplace, WeChat SDK wrapper, account system,
ads, monetization, analytics service, mobile app shell, publishing workflow,
game-specific rules engine, water-sort solver, match-3 solver, galgame script
format, or content authoring workflow.

The helper only assembles existing browser input runtime bridges for a scene.
Gameplay rules, input action definitions, HUD layout, audio cue choices, asset
data, scene routing, and example-specific cleanup remain downstream-owned.

## Done When

This slice is done when:

- the helper is exported from `/framework`
- tests cover keyboard target injection, bridge bundle attach/detach, destroy
  cleanup, and missing `InputSystem` diagnostics
- examples consume the helper through package-style imports
- package boundary docs record this as quick-start game kit work
- `npm run check`, `npm test`, `npm run build:example`,
  `npm run verify:package`, and `npm audit --audit-level=moderate` pass
