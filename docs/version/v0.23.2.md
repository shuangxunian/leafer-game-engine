# leaferGame v0.23.2

## Version Positioning

`v0.23.2` is the second implementation pass in the `0.23.x` stage:

**Camera Bounds and Follow Clamping Primitives**

`v0.23.1` added copied camera viewport state and deterministic world/viewport coordinate conversion. This version adds reusable camera bounds primitives so downstream games can keep the visible world area inside a declared world rectangle during manual movement and follow behavior.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.23.2` does not add visual scene editors, camera track/timeline editors, cinematic sequencing, keyframe authoring, editor handles/gizmos, scene graph mutation UI, camera authoring tools, save/load workflows, or content production tooling.

It only hardens runtime camera movement constraints.

---

## Goals

After `v0.23.2`, the project supports:

- a small camera bounds data contract
- setting, clearing, and inspecting copied camera bounds
- clamping manual `moveTo(...)` calls to bounds
- clamping follow target results to bounds
- deterministic behavior when the visible viewport is larger than the declared bounds
- Node-safe tests for manual and follow clamping semantics
- public docs preserving the runtime-package/not-editor boundary

---

## Scope

This version includes:

- plain TypeScript camera bounds data
- `CameraSystem` bounds helpers
- clamping based on the current zoom and render viewport size
- tests for manual movement, follow movement, zoom changes, and small bounds
- public docs and package-boundary coverage

This version does not include:

- camera smoothing, shake, damping, or transitions
- dead zones
- camera pathing or rails
- read-only tooling snapshots
- dodge-blocks camera consumption
- visual camera controls
- editor or authoring workflows

---

## Semantics

- Camera `x` / `y` remain the world-space center point.
- Bounds describe the world-space rectangle that the visible viewport should stay inside.
- `moveTo(...)` should store the clamped camera center when bounds are active.
- Follow behavior should also resolve to a clamped camera center when bounds are active.
- When visible viewport dimensions are larger than the bounds, the camera should center on the bounds axis instead of producing an invalid clamp interval.
- Bounds snapshots must be copied so callers cannot mutate internal camera state.
