# leaferGame v0.24.5

## Version Positioning

`v0.24.5` closes the `0.24.x` playable 2D game kit stage:

**Playable Example Closeout And Package Docs**

This version should make the current example read like a downstream browser game that consumes the engine package, not like a pile of local demo glue. The goal is to close the 4399-style mini-game proof point by documenting what is already possible, exposing a small read-only gameplay snapshot from the example system, and keeping the product boundary focused on runtime/gameplay consumption.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.24.5` does not add an editor, scene builder, visual gameplay debugger, map editor, menu designer, asset manager, save/load project workflow, leaderboard backend, monetization SDK, or publishing pipeline.

It only closes the current playable example stage by making runtime state and package-facing docs easier to inspect.

---

## Goals

After `v0.24.5`, the project should support:

- a concise example gameplay snapshot for `dodge-blocks`
- package-boundary coverage that keeps the playable example direction visible
- README wording that explains the project can now support a simple browser mini-game prototype
- docs that clearly separate engine/package capability from future editor or content-production work
- a clean endpoint for `0.24.x` before choosing the next stage

---

## Scope

This version includes:

- `DodgeGameSystem.getGameplaySnapshot()` for read-only example gameplay state
- example/docs coverage for phase, score, best score, hazard count, survival time, and active state
- README and public API status updates for `v0.24.5`
- roadmap and `v0.24.0` closeout notes
- package-boundary tests that lock the closeout into docs and example source

This version does not include:

- generic framework-level gameplay snapshot APIs
- new gameplay modes
- game persistence or browser storage
- online leaderboard services
- editor UI, visual scene mutation, or authoring workflows
- tile map editing or asset management

---

## Semantics

- The snapshot is read-only copied data from the example gameplay system.
- `phase` comes from the existing framework `GameFlow`.
- `score`, `bestScore`, and `survivalTimeSeconds` reflect the current local run.
- `hazardCount` reports current runtime hazards owned by the example system.
- `isGameplayActive` mirrors whether gameplay update should advance.
- This is an example-level closeout helper, not a new public framework primitive.
