# leaferGame v0.8.0

## Version Positioning

`v0.8.0` starts the next stage:

**Resource Loading And Asset Pipeline Sprint**

The previous stage made runtime inspector tooling interactive. This stage returns to engine package fundamentals by making asset handling more complete than synchronous metadata registration.

---

## Why This Stage

Current asset support can declare and register sprite metadata through `AssetManifest`, but it does not yet model real loading workflow:

- no async asset loading abstraction
- no loading status snapshot
- no distinction between registered metadata and loaded resource readiness
- no partial success result for manifests
- no browser image loading adapter yet

For a reusable frontend 2D game engine package, asset loading is a core engine capability. It should be explicit, testable, and independent from any single example.

---

## Stage Goals

`0.8.x` should move the project toward:

- async asset loading primitives
- stable load status/result data
- manifest loading that can report partial success
- browser image loading integration
- tooling/debug visibility for asset load state
- sample integration that still keeps examples as consumers

## Scope Boundaries

This stage should include:

- framework-level loader interfaces
- deterministic load results
- tests for success, failure, and skipped assets
- browser image loader as an adapter-style helper
- runtime tooling visibility where useful

This stage should not include:

- bundler plugin integration
- remote CDN policy
- texture atlas packing
- audio loading
- streaming assets
- editor UI, asset browser UI, or visual authoring workflow

---

## Done When

By the end of `0.8.x`:

- assets can be registered and loaded through explicit APIs
- load status can be inspected programmatically
- manifest loading can report success/failure without hiding errors
- browser examples can use the loading path
- `npm run check`, `npm test`, and `npm run build:example` pass

---

## Relation to Previous Stages

- `0.5.x` introduced asset manifests and data-driven content primitives.
- `0.6.x` and `0.7.x` made runtime state easier to inspect.
- `0.8.x` should make assets more production-ready as part of the engine package itself.
