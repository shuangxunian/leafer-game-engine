# leaferGame v0.22.2

## Version Positioning

`v0.22.2` is the second implementation pass in the `0.22.x` stage:

**Audio Playback System Draining Integration**

`v0.22.1` introduced a Node-safe playback adapter contract and deterministic drain helper. This version should make that contract easier to use from scenes by adding an optional framework system that drains `AudioRuntimeState` operations into an injected adapter during scene updates.

---

## Product Boundary

This remains frontend 2D game engine package work.

`v0.22.2` does not add Web Audio playback, HTMLAudioElement playback, decoding, streaming audio, asset loading, waveform UI, timeline editing, mixer controls, volume sliders, asset browser UI, audio authoring, project persistence, or content production workflows.

It only adds opt-in scene/system draining integration for an injected playback adapter.

---

## Goals

After `v0.22.2`, the project should support:

- an `AudioPlaybackSystem` that can be installed into a `Scene`
- an `addAudioPlayback(...)` helper
- a `getAudioPlayback(...)` helper
- deterministic update-driven draining of pending audio runtime operations
- copied playback result history for tests and later read-only observability
- explicit configuration for clearing or preserving drained operation records
- Node-safe tests using fake adapters

---

## Scope

This version should include:

- framework-level system integration around the existing adapter/drain helper
- package exports from `/framework`
- tests for scene-owned audio playback draining
- tests for injected audio state and adapter failure result history
- public docs clarifying this is system integration, not browser playback

This version should not include:

- Web Audio implementation
- HTMLAudioElement implementation
- audio asset loading or decoding
- browser permissions/autoplay handling
- spatial audio
- tooling panel changes
- dodge-blocks playback consumption
- editor/mixer/authoring UI

---

## Semantics

- The playback system should be opt-in.
- It should accept an injected `AudioPlaybackAdapter`.
- It should drain a known `AudioRuntimeState`, or use the scene-owned `AudioRuntimeSystem` state when no state is injected.
- It should run during update and preserve deterministic operation order.
- It should expose copied last-drain result history.
- Adapter failures should remain result data, not hidden side effects.
- Tests should run in Node with no DOM or media APIs.

---

## Done When

- A scene can install audio runtime + playback systems and drain operations during update.
- A fake adapter receives operations in deterministic order.
- Playback result history exposes sequence, operation type, status, and optional error text.
- Public exports include the playback system and helpers.
- Public docs preserve the runtime-package/not-audio-editor boundary.
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass.
