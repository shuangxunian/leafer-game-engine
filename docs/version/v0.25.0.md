# leaferGame v0.25.0

## Version Positioning

`v0.25.0` starts the next stage:

**Second Playable Example Sprint**

`0.24.x` proved that the engine package can support one simple 4399-style browser mini-game loop through `examples/dodge-blocks`. The next stage should check whether those primitives are truly reusable by adding a second playable example with a different gameplay shape.

This stage answers the product question: are movement, input actions, actor composition, HUD helpers, tile/map consumption, collision, audio intent/playback, GameFlow, browser runtime, and read-only tooling usable outside a single dodge-blocks sample?

---

## Why This Stage

The project now has a useful first playable loop:

- semantic input actions and keyboard/pointer button bridges
- normalized movement helpers
- actor template composition
- runtime HUD text helpers
- tile map layer view consumption
- collision queries
- sprite animation and asset loading
- audio intent and optional browser playback
- GameFlow and scene lifecycle helpers
- read-only tooling snapshots and browser panel

The current gap is proof of reuse. A single example can accidentally hide example-specific assumptions. A second example should force the package-facing APIs to stand on their own without turning this repository into a game gallery or editor product.

This remains frontend 2D game engine package work, not an editor, template marketplace, visual scene builder, asset manager, level authoring tool, or content publishing workflow.

---

## Stage Goals

`0.25.x` should move the project toward:

- a second runnable browser example with a different core loop from dodge-blocks
- reuse of existing framework primitives instead of copy-pasting one-off local helpers
- example docs that explain which package APIs are being consumed
- package-boundary tests that prevent examples from importing `src` directly
- shared example boot/import conventions where useful
- README wording that explains the engine now has more than one downstream-style consumer

## Candidate Example Shape

The second example should be intentionally small. A good candidate is a collection/chase loop:

- player moves inside a bounded playfield
- collectibles spawn at runtime
- hazards or timers create pressure
- score and status use HUD helpers
- phase uses `GameFlow`
- player/collectible/hazard templates consume actor composition where practical
- collision queries determine pickup or failure
- optional audio cues remain semantic runtime intent, not authoring UI

The exact theme can change, but the example should validate a different gameplay loop from falling-block dodging.

---

## Scope Boundaries

This stage should include:

- a second example under `examples/`
- package-style imports from `@shuangxunian/leafer-game-engine`
- browser build coverage
- example README coverage
- package-boundary tests for import discipline and non-editor positioning
- small reusable improvements only when the second example exposes real friction

This stage should not include:

- a visual example launcher product
- editor UI
- drag-and-drop scene authoring
- map editing or save/load project files
- asset marketplace or template gallery
- online services, leaderboards, accounts, monetization, or publishing flows
- broad framework abstractions before two examples prove the need

---

## Done When

By the end of `0.25.x`:

- at least two examples prove package-style consumption
- the second example has a playable loop distinct from dodge-blocks
- examples remain consumers, not engine internals
- README describes the project as a small engine package with multiple downstream-style examples
- package-boundary tests cover the new example and stage docs
- `npm run check`, `npm test`, `npm run build:example`, and `npm run verify:package` pass
